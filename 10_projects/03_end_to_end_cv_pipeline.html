10.3 Project: End-to-End CV Pipeline

Build a portfolio-grade computer vision project: dataset, training, evaluation, error analysis, export, and FastAPI inference endpoint.

TL;DR

The goal is not just a good model. The goal is a reproducible CV system with clear data assumptions, metrics, failure cases, and a deployable inference contract.

Prerequisites5.9 Detection, 5.10 Segmentation, 9.2 Serving
Estimated time12-24 hours
Job relevanceHigh
Portfolio valueHigh if documented well

Deliverables

  • Dataset card: source, labels, splits, class imbalance, known bias.
  • Training script with config file and deterministic split.
  • Metrics report: mAP/IoU/F1, latency, confusion/error cases.
  • Model export artifact and inference script.
  • FastAPI endpoint with request/response examples.
  • README with setup, results table, and demo images.

System pipeline

raw images -> validation -> annotation format -> training -> evaluation -> export -> API -> monitoring

What to notice: most real CV bugs happen before and after training: wrong labels, wrong coordinate format, broken preprocessing, inconsistent resize, or serving-time mismatch.

Error analysis

  • Sort false positives by confidence: what patterns fool the model?
  • Sort false negatives by object size and lighting.
  • Check performance per class, not only global score.
  • Measure latency with realistic image sizes and batch size 1.

Разбор ключевой формулы

pipeline quality = data quality + model quality + serving quality
Что означают символы
data qualityданные и разметканасколько dataset чистый, согласованный и релевантный задаче.
model qualityкачество моделиmAP, Dice, F1 или другая выбранная метрика.
serving qualityкачество доставкиlatency, robustness и совпадение preprocessing на inference с train.
Интуиция: end-to-end CV pipeline ломается чаще на данных и serving, чем на самой архитектуре.
Числовой пример: даже при хорошем detector serving-time resize mismatch может резко ухудшить mAP в проде.
