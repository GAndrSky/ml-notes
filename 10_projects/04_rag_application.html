TL;DR

A RAG app is not "LLM plus vector database". It is an information retrieval system plus a generation layer. Most quality problems come from chunking, retrieval recall, reranking, context assembly, and evaluation.

Pipeline

documents -> chunks -> embeddings -> vector index -> top-k retrieval -> context -> LLM answer + citations

What to notice: if the answer is wrong, first check whether the correct chunk was retrieved. Generation cannot fix missing evidence.

Retrieval objective

score(q, d_i) = cosine(embed(q), embed(d_i))
What the symbols mean
qqueryUser question or rewritten search query.
d_ichunkOne candidate document fragment.
embedembedding modelMaps text into a vector space where similarity approximates relevance.
cosinesimilarityCompares vector directions, usually after normalization.
Intuition: retrieval finds chunks whose meaning points in the same direction as the question.
Analogy: library search returns pages most semantically aligned with your question.

Runnable toy retrieval

# What it does: toy vector retrieval with cosine similarity
# Input shapes: query=(d,), docs=(N,d)
# Output shape: top indices=(k,)
import numpy as np

def normalize(x):
    return x / (np.linalg.norm(x, axis=-1, keepdims=True) + 1e-9)

docs = np.array([[1,0], [0,1], [0.8,0.2]], dtype=float)
query = np.array([1,0.1], dtype=float)
scores = normalize(docs) @ normalize(query[None, :]).T
top = np.argsort(scores[:, 0])[::-1][:2]
print(top, scores[top, 0])
# Gotchas: semantic embeddings are not magic; evaluate recall on real questions.

Common mistakes

  • Chunking by fixed length without preserving section boundaries.
  • Using only answer quality while ignoring retrieval recall.
  • Stuffing too much context and reducing answer focus.
  • Not storing metadata, making citations impossible.

Пошагово RAG query pipeline
1 /
Шаг 01

Embed the query

User query arrives. Embed it with the same model used to embed documents during indexing. Using a different model produces incompatible vector spaces where cosine similarity is meaningless.

Шаг 02

Vector search

Retrieve top-K chunks by cosine similarity from the vector index. Typical K=5-20. More chunks increase recall but also increase prompt length and LLM cost. Start with K=10 and tune based on retrieval recall metrics.

Шаг 03

Reranking

Pass query + retrieved chunks to a cross-encoder, rerank by relevance score, and keep the top 3. Cross-encoders are slower but much more accurate than bi-encoders for relevance judgment because they process query and document together.

Шаг 04

Context assembly

Prepend the retrieved chunks to the prompt with source attribution metadata (document name, page, section). Include source IDs so the generator can cite them. Keep total context within the LLM's context window minus the expected answer length.

Шаг 05

Generation with citations

LLM generates an answer grounded in the provided context. Return both the answer and source citations. Evaluate groundedness: does the answer make claims supported by the retrieved chunks? Unsupported claims are hallucinations.

Проверь себя

RAG application

  1. Why must you use the same embedding model for documents and queries?

    Embedding models learn a specific vector space where similar texts are geometrically close. Different models learn different spaces — a vector from model A and a vector from model B do not share a coordinate system. Cosine similarity between them is meaningless and retrieval becomes random. Always index and query with the same model version.

  2. What is the difference between a bi-encoder and a cross-encoder in retrieval?

    A bi-encoder embeds query and document independently and computes similarity via dot product — fast but less accurate. A cross-encoder takes query and document as a single concatenated input and produces a relevance score — much more accurate because it can model interactions, but cannot be pre-indexed and is 100-1000x slower. Use bi-encoder for recall at scale, cross-encoder for reranking top candidates.

  3. How do you evaluate whether a RAG system is hallucinating?

    Check groundedness: does every claim in the answer appear in the retrieved context? Use LLM-as-judge to verify each sentence against the context, or use an NLI model to check entailment from context to answer. The RAGAS faithfulness metric automates this. Also check retrieval first: if the correct chunk was not retrieved, the answer will be wrong regardless of the LLM quality.

Что ломается без этого

Using different embedding models for indexing and querying produces random retrieval — cosine similarity between different embedding spaces is meaningless. The system returns documents with high vector similarity that have no semantic relationship to the query. All downstream quality metrics degrade for reasons that are hard to diagnose unless you know to check embedding model consistency first.

Проверь себя

Вопросы для самопроверки

  1. Как выбрать embedding model для RAG системы?

    Критерии: (1) размерность — больше не всегда лучше (768 vs 1536), (2) benchmark на domain-specific задачах (MTEB leaderboard), (3) поддержка русского языка при необходимости, (4) inference speed, (5) максимальная длина контекста. BAAI/bge и E5 — сильные общие модели.

  2. Что такое chunking overlap и зачем он нужен?

    Overlap: соседние chunks делят несколько предложений. Без overlap релевантная информация может оказаться на стыке chunks — оба chunk частично «половинчаты» и ни один не релевантен полностью. Overlap 10-20% предотвращает это, добавляя контекст на краях.

  3. Как оценить качество RAG системы?

    Метрики: (1) Retrieval: precision@k, recall@k, nDCG (насколько релевантны найденные chunks). (2) Generation: faithfulness (нет ли hallucinations вне retrieved context), answer relevance, context relevance. Ragas — библиотека для автоматической оценки RAG по этим метрикам.

  4. Что такое HyDE (Hypothetical Document Embeddings) и как он улучшает retrieval?

    HyDE: вместо поиска по исходному запросу генерируем LLM гипотетический ответ, затем ищем документы по embedding этого ответа. Гипотетический ответ семантически ближе к реальным документам, чем короткий вопрос. Улучшает recall, особенно для абстрактных или сложных запросов.

Как закрепить тему: RAG Приложение

Блок связывает теорию с практикой: где применять идею, какие ошибки проверять и как развиваться дальше.

Когда применять

  • Начинай с BM25 retrieval + GPT-4 generation — простой baseline для оценки потолка.
  • Ragas или LLM-as-judge обязателен для оценки faithfulness.
  • Мониторь, какие запросы не находят релевантных документов — это сигнал о coverage gap.

Типичные ошибки

  • Не разделять retrieval и generation ошибки при дебаггинге.
  • Не тестировать на adversarial запросах (вне knowledge base, противоречивые вопросы).
  • Не учитывать context window при выборе chunk_size × top_k.

Что тюнить / проверить

  • chunk_size: 200-500 слов для большинства задач.
  • top_k: 3-5 для начала, увеличивай при низком recall.
  • Temperature LLM: снижай для factual задач (0.1-0.3).
