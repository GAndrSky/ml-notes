10.2 Project: Fine-tune an LLM with LoRA

Build a complete parameter-efficient fine-tuning pipeline: prepare an instruction dataset, train LoRA adapters, evaluate behavior, export artifacts, and document failure modes.

TL;DR

LoRA freezes the base model and trains small low-rank update matrices. This project teaches the practical workflow: data quality, prompt format, adapter placement, hyperparameters, evaluation, and safe export.

Project scope

Deliverables
  • A small instruction dataset in a documented format.
  • A training script using Transformers, Datasets, and PEFT.
  • A saved LoRA adapter with config and tokenizer.
  • An evaluation notebook with before/after examples.
  • A README explaining limits, cost, and failure cases.
Scope control
  • Start with a small model and a tiny dataset.
  • Do not claim the model learned facts from 50 examples.
  • Measure style and task-following changes separately from knowledge.
  • Keep a held-out validation set of prompts.

Core idea

W' = W + DeltaW, where DeltaW = B A and rank(DeltaW) << rank(W)
What the symbols mean
Wfrozen base weightOriginal pretrained model matrix, not updated during LoRA training.
DeltaWlearned updateTask-specific change added to the base weight.
Adown projectionSmall matrix that maps from original dimension into rank r.
Bup projectionSmall matrix that maps from rank r back to output dimension.
rLoRA rankControls adapter capacity; common values are 4, 8, 16, 32, 64.
Intuition: instead of changing all model weights, learn a small structured correction in the directions that matter for the task.
Analogy: keep the whole book unchanged and add a thin task-specific annotation layer.
trainable_params = r * (d_in + d_out) instead of d_in * d_out
Numeric example
d_ininput widthFor example 4096 in a transformer projection.
d_outoutput widthFor example 4096 in the same projection.
rrankIf r=16, trainable params are 16*(4096+4096)=131k instead of 16.7M.
Intuition: the trainable part becomes tiny compared with the original matrix.
Practical meaning: you can fine-tune larger models on smaller GPUs because optimizer state is stored only for adapters.

Milestone 1 - dataset format

For instruction tuning, data format often matters more than rank or learning rate. Keep examples consistent and aligned with how you will prompt the model at inference.

# What it does: creates a tiny instruction dataset in chat format.
# Input shapes: list[dict] with instruction/input/output strings
# Output shapes: Hugging Face Dataset with one text column
from datasets import Dataset

rows = [
    {
        "instruction": "Explain gradient descent to a beginner.",
        "input": "",
        "output": "Gradient descent repeatedly moves parameters in the direction that reduces loss."
    },
    {
        "instruction": "Convert the note into three bullet points.",
        "input": "Adam keeps moving averages of gradients and squared gradients.",
        "output": "- Adam tracks gradient direction\n- Adam tracks gradient scale\n- Adam adapts step sizes per parameter"
    },
]

def format_example(row):
    return {
        "text": (
            "### Instruction:\n" + row["instruction"].strip() +
            "\n\n### Input:\n" + row["input"].strip() +
            "\n\n### Response:\n" + row["output"].strip()
        )
    }

dataset = Dataset.from_list([format_example(row) for row in rows])
print(dataset[0]["text"])

# Gotchas:
# 1. Keep train and inference prompt formats consistent.
# 2. Do not mix contradictory answer styles in a tiny dataset.

Milestone 2 - configure LoRA

# What it does: wraps a causal LM with LoRA adapters.
# Input shapes: tokenized batch input_ids=(B,L), labels=(B,L)
# Output shapes: model returns loss=scalar and logits=(B,L,V)
from transformers import AutoModelForCausalLM, AutoTokenizer
from peft import LoraConfig, get_peft_model, TaskType

model_name = "TinyLlama/TinyLlama-1.1B-Chat-v1.0"
tokenizer = AutoTokenizer.from_pretrained(model_name)
tokenizer.pad_token = tokenizer.eos_token

model = AutoModelForCausalLM.from_pretrained(model_name, device_map="auto")

lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,
    r=16,
    lora_alpha=32,
    lora_dropout=0.05,
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj"],
)

model = get_peft_model(model, lora_config)
model.print_trainable_parameters()

# Gotchas:
# 1. target_modules depend on model architecture; inspect module names if this fails.
# 2. Larger r increases capacity but also memory and overfitting risk.

Milestone 3 - train

# What it does: trains LoRA adapters with Hugging Face Trainer.
# Input shapes: dataset["text"] tokenized to input_ids=(B,L)
# Output shapes: saved adapter files in output_dir
from transformers import TrainingArguments, Trainer, DataCollatorForLanguageModeling

def tokenize(batch):
    return tokenizer(
        batch["text"],
        truncation=True,
        max_length=512,
        padding=False,
    )

tokenized = dataset.map(tokenize, batched=True, remove_columns=["text"])
collator = DataCollatorForLanguageModeling(tokenizer=tokenizer, mlm=False)

args = TrainingArguments(
    output_dir="outputs/lora-ml-notes-demo",
    per_device_train_batch_size=1,
    gradient_accumulation_steps=8,
    learning_rate=2e-4,
    num_train_epochs=3,
    logging_steps=1,
    save_strategy="epoch",
    fp16=False,
    bf16=True,
    report_to=[],
)

trainer = Trainer(
    model=model,
    args=args,
    train_dataset=tokenized,
    data_collator=collator,
)
trainer.train()
model.save_pretrained("outputs/lora-adapter")
tokenizer.save_pretrained("outputs/lora-adapter")

# Gotchas:
# 1. If bf16 is unsupported, set bf16=False and use fp16 only with care.
# 2. Tiny datasets can quickly overfit style without improving task ability.

Milestone 4 - evaluate before and after

Do not evaluate only by looking at one pretty answer. Build a small fixed set of prompts and compare base model vs adapted model using the same decoding settings.

Qualitative eval

  • Does the model follow the target format?
  • Does it avoid hallucinating unsupported details?
  • Does it preserve useful base-model ability?
  • Does it become too repetitive or too short?

Quantitative eval

  • Held-out loss on validation prompts.
  • Exact format compliance for structured outputs.
  • Human preference labels for before/after answers.
  • Latency and memory if adapter is used in serving.
# What it does: generates a response from the LoRA-adapted model.
# Input shapes: prompt tokens input_ids=(1,L)
# Output shapes: generated tokens=(1,L_new)
import torch

prompt = "### Instruction:\nExplain LoRA in two sentences.\n\n### Input:\n\n### Response:\n"
inputs = tokenizer(prompt, return_tensors="pt").to(model.device)

with torch.no_grad():
    generated = model.generate(
        **inputs,
        max_new_tokens=80,
        do_sample=False,
        temperature=None,
        pad_token_id=tokenizer.eos_token_id,
    )

print(tokenizer.decode(generated[0], skip_special_tokens=True))

# Gotchas:
# 1. Use identical decoding settings when comparing base and adapted models.
# 2. Greedy decoding is stable for tests; sampling is useful for product behavior checks.

Milestone 5 - export and document

For a portfolio project, the README matters. It should make the experiment reproducible and honest.

README checklist

  • Base model and license.
  • Dataset source, size, filtering, and examples.
  • Hardware, training time, and peak memory.
  • Hyperparameters and adapter target modules.
  • Before/after examples and failure cases.

Export options

  • Save adapter only for PEFT loading.
  • Merge adapter into base weights for simpler inference.
  • Quantize for deployment only after checking quality.
  • Keep tokenizer and prompt template versioned.

Exercises

Do not skip exercises. LoRA projects fail most often because of poor data, weak evaluation, or unrealistic claims.

Foundational - data

Create 50 high-quality examples

Build a tiny instruction dataset for one narrow behavior, such as converting ML notes into interview answers.

Show solution

Use one prompt format, consistent output style, and a validation split. Keep examples narrow enough that you can judge success manually.

Intermediate - training

Run LoRA with two ranks

Train with r=8 and r=32. Compare train loss, validation behavior, and output style.

Show solution

Higher rank can fit more task-specific behavior but can overfit small data. Use the held-out prompts to decide, not train loss alone.

Intermediate - evaluation

Build a before/after eval table

For 10 held-out prompts, compare base and adapted answers side by side with notes.

Show solution

Track format compliance, correctness, concision, hallucination, and whether the answer follows the intended style.

Hard - ablation

Change target modules

Train one adapter on q/v only and one on q/k/v/o. Compare behavior and trainable parameter count.

Show solution

More target modules increase capacity and cost. On tiny data, the difference may be hard to measure unless the task is clear and evaluation is fixed.

Hard - deployment

Serve the adapter locally

Load the base model plus adapter and expose one `/generate` endpoint with FastAPI.

Show solution

Keep model loading outside the request handler, validate max_new_tokens, log latency, and return both answer and generation settings.

Learning checkpoint

  • Can you explain why LoRA reduces trainable parameters?
  • Can you compute trainable parameters for a LoRA-adapted matrix?
  • Can you describe the full pipeline from raw examples to saved adapter?
  • Can you evaluate whether fine-tuning improved behavior or only changed style?
  • Can you name the main failure modes of tiny-data LoRA?
