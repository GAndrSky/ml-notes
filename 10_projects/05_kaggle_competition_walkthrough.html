10.5 Project: Kaggle Competition Walkthrough

A practical competition workflow: establish a baseline, build reliable validation, engineer features, compare models, ensemble carefully, and submit without fooling yourself.

Зачем это знать

Kaggle — лучшая площадка для практики ML на реальных данных. Понимание workflow соревнования ускоряет разработку ML-моделей в любом контексте.

TL;DR

Kaggle success is less about trying many models randomly and more about validation discipline. If your local validation does not predict leaderboard movement, every improvement is suspect.

Workflow

  1. Read metric and submission format first.
  2. Build a dumb baseline.
  3. Create leakage-safe validation.
  4. Run EDA focused on target, missingness, groups, time, and leakage.
  5. Train simple robust models before complex ensembles.
  6. Track experiments and compare local score to public leaderboard.
  7. Ensemble only models with diverse errors.

Validation is the project

trust = correlation(local_cv_score, leaderboard_score)

If local CV improves but leaderboard gets worse repeatedly, the validation split is probably wrong. For time-series data, use time splits. For user/item data, use group splits. For duplicated entities, split by entity, not row.

Runnable validation skeleton

# What it does: leakage-safe grouped cross-validation skeleton
# Input shapes: X=(N,d), y=(N,), groups=(N,)
# Output shape: fold scores=(K,)
import numpy as np
from sklearn.model_selection import GroupKFold
from sklearn.metrics import mean_squared_error
from sklearn.ensemble import RandomForestRegressor

X = np.random.randn(100, 5)
y = X[:, 0] * 2 + np.random.randn(100) * 0.1
groups = np.repeat(np.arange(20), 5)

scores = []
for train_idx, valid_idx in GroupKFold(n_splits=5).split(X, y, groups):
    model = RandomForestRegressor(n_estimators=50, random_state=0)
    model.fit(X[train_idx], y[train_idx])
    pred = model.predict(X[valid_idx])
    scores.append(mean_squared_error(y[valid_idx], pred, squared=False))
print(scores, np.mean(scores))
# Gotchas: choose split strategy before feature engineering; never fit preprocessing on validation data.

Feature engineering checklist

  • Missing-value indicators for informative missingness.
  • Target-safe aggregations computed inside folds.
  • Time features and lag features for temporal data.
  • Count/frequency encodings for categorical variables.
  • Text/image embeddings if competition data is multimodal.

Common mistakes

  • Using the public leaderboard as validation.
  • Leaking target through global target encoding or preprocessing.
  • Changing many things at once and not knowing what helped.
  • Overfitting to the public leaderboard with too many submissions.

Пошагово Competition strategy
1 /
Шаг 01

EDA first

Understand the data, look for leaks, and check train/test distribution shift before writing any model code. Target leakage found early saves days of wasted work. Distribution shift explains why a strong local model underperforms on the leaderboard.

Шаг 02

Leakage-safe validation

Always use stratified KFold matching the competition metric. For time-series use time-ordered splits. For grouped data (same entity in multiple rows) use GroupKFold. Never look at test labels or use test statistics in feature engineering. Fix the validation scheme before touching features.

Шаг 03

Baseline + feature engineering

Tabular: gradient boosting baseline first. NLP: pretrained transformer. CV: pretrained CNN or ViT. Get a number on the leaderboard before engineering features. Feature engineering on a bad baseline wastes time; do it after confirming the validation scheme is trusted.

Шаг 04

Ensembling

Average predictions from diverse models. Correlation below 0.95 between model predictions gives useful ensemble gains. Diversity comes from different algorithms (GBM vs NN vs linear), different feature sets, and different random seeds. Ensembling identical models adds no benefit.

Шаг 05

Late-stage: pseudo-labeling

Pseudo-labeling on high-confidence test predictions can give a final boost in the last days. Train on original training data plus confident test predictions treated as labeled. Only use test examples where the model is very confident (top 10-20% by confidence). Validate that CV score improves before submitting.

Проверь себя

Kaggle competition strategy

  1. What is target leakage and how do you detect it?

    Target leakage occurs when features contain information about the target that would not be available at prediction time. Detection: (1) check if any feature has near-perfect correlation with the target, (2) check if removing suspicious features causes a dramatic score drop, (3) check feature timestamp — was it recorded before or after the target event? (4) compare feature distributions between train/test — a feature absent from test is suspicious.

  2. Why is stratified cross-validation important for imbalanced competitions?

    Random splits may put almost all positive examples in one fold, making other folds useless for evaluating minority-class performance. Stratified KFold maintains the class ratio in each fold, giving stable and representative evaluation. For very imbalanced data (1% positive) a random split could produce a fold with 0 positives, making fold evaluation impossible.

  3. When does ensembling improve the leaderboard score?

    Ensembling helps when models make different errors — low prediction correlation means their mistakes do not overlap. Averaging diverse models reduces variance. It does not help when all models are similar (same algorithm, same features, different seeds only). Best gains come from ensembling fundamentally different model types: GBM plus NN plus linear model each bring different inductive biases.

Что ломается без этого

Validating on a random split instead of a time-ordered split for time-series data causes severe overfitting to future data — CV scores will be much better than the public leaderboard. The model learns future patterns that are not available at prediction time. This is the most common reason strong local CV scores fail to translate to leaderboard performance in time-series competitions.

Проверь себя

Вопросы для самопроверки

  1. Какой стратегии придерживаться в первые дни Kaggle соревнования?

    EDA → baseline submission (простая модель) → понять gap до lb → feature engineering → кросс-валидация. Не тюни гиперпараметры раньше, чем построишь надёжную CV. Часто baseline XGBoost/LightGBM + базовые признаки даёт 80% от финального результата.

  2. Почему Local CV важнее Public Leaderboard при разработке?

    Public LB использует только часть тестовых данных (обычно 20-30%). Переоптимизация под него приводит к shake-up на private LB (финальном). Надёжная кросс-валидация коррелирует с private LB. Правило: доверяй CV, а не public LB.

  3. Что такое «shake-up» на Kaggle и почему он происходит?

    Shake-up — изменение позиций в рейтинге при переходе с public на private LB. Происходит из-за переобучения под public LB (маленькая выборка), случайности в оценке или overfitting к особенностям конкретного сплита. Защита: ансамблирование, стабильная CV, не оптимизировать под public aggressively.

  4. Как строить финальный ансамбль для Kaggle?

    Стратегии: (1) усреднение (soft voting) разных типов моделей (GBM + NN + linear), (2) rank averaging — усреднение рангов предсказаний, (3) stacking — meta-learner на OOF предсказаниях. Ключевое: diverse models с низкой корреляцией ошибок дают лучший ансамбль, чем 10 похожих моделей.

Как закрепить тему: Kaggle Competition

Блок связывает теорию с практикой: где применять идею, какие ошибки проверять и как развиваться дальше.

Когда применять

  • Фиксируй CV схему и seed с первого дня — без этого нельзя сравнивать эксперименты.
  • Читай discussion и notebooks лидеров — domain knowledge часто важнее ML.
  • Финальный сабмит: два разных ансамбля на случай shake-up.

Типичные ошибки

  • Оптимизировать под public LB без надёжной CV — гарантированный shake-up.
  • Тратить всё время на feature engineering без проверки baseline.
  • Делать единственный финальный submission — всегда сохраняй разные варианты.

Что тюнить / проверить

  • CV схема: надёжность важнее correlation с public LB.
  • Ensemble weights: оптимизируй на holdout, не на public LB.
  • Feature selection: permutation importance на OOF предсказаниях.
