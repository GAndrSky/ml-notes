2.18 Несбалансированные классы

Почему accuracy ломается, class weights, over/under-sampling, SMOTE и какие метрики реально важны при imbalance.

Почему imbalance — это отдельная проблема

Во многих реальных задачах классы распределены неравномерно. Например, мошенничество, дефекты, редкие заболевания, отток клиентов, аварии, спам, фрод, отказ оборудования — всё это обычно редкие события.

Пример: Class 0 = 99% Class 1 = 1% Модель, которая всегда говорит "0", получит accuracy = 99%
Интуиция. Когда положительный класс редкий, модель может почти игнорировать его и всё равно показывать высокую “среднюю” точность. Это делает стандартный взгляд на качество обманчивым.
В задачах с imbalance вопрос обычно не в том, “насколько хорошо модель угадывает большинство”, а в том, насколько хорошо она находит редкий, но важный класс.

2.18.1 Почему accuracy ломается

Accuracy считает долю всех правильных ответов:

Accuracy = (TP + TN) / (TP + TN + FP + FN)

Но если класс 0 сильно доминирует, TN может быть огромным, и accuracy будет большой даже у бесполезной модели.

Поэтому accuracy хороша только тогда, когда классы более-менее сбалансированы и цена ошибок примерно одинакова.
Для imbalance accuracy часто маскирует провал по minority class.

2.18.2 Class weights

Один из самых простых способов бороться с imbalance — сделать ошибки на minority class дороже.

Идея: loss на объектах редкого класса умножается на больший вес Weighted loss: L = Σ_i w_{y_i} · loss_i
Это заставляет модель сильнее реагировать на ошибки по minority class.
Но слишком сильные веса могут ухудшить precision и сделать модель слишком “нервной”.
Интуиция. Если положительный класс редкий, без весов модель учится в мире, где ошибаться по редкому классу “дёшево”. Class weights меняют эту экономику.

2.18.3 Oversampling и undersampling

Oversampling

Увеличиваем minority class, дублируя или синтетически создавая дополнительные объекты.

Плюс: модель лучше “видит” minority class.
Минус: простой duplicate oversampling может усиливать overfitting.

Undersampling

Уменьшаем majority class, случайно выбрасывая часть его объектов.

Плюс: баланс становится лучше, обучение может стать быстрее.
Минус: можно выбросить полезную информацию о majority class.
Oversampling: minority ↑ Undersampling: majority ↓ Обе техники меняют обучающую выборку, но не меняют реальное распределение мира.

2.18.4 SMOTE

SMOTE (Synthetic Minority Over-sampling Technique) — это не просто дублирование minority объектов. Он генерирует новые synthetic samples между существующими объектами minority class.

Упрощённая идея SMOTE: берём minority point x_i находим его minority-neighbor x_j создаём новый объект: x_new = x_i + λ(x_j - x_i), где λ ∈ [0,1]
В отличие от простого oversampling, SMOTE создаёт новые точки внутри области minority class, а не просто копирует старые.
Но SMOTE тоже не универсален. Если minority class сложный, многомодальный, шумный или пересекается с majority, synthetic points могут появляться в неудачных местах.
SMOTE особенно полезен, когда minority class слишком редкий, но его геометрия в пространстве признаков всё же более-менее осмысленна.

2.18.5 Какие метрики важны при imbalance

Метрика Смысл Почему полезна при imbalance
Recall Сколько положительных нашли Показывает, насколько модель пропускает minority class
Precision Насколько чисты найденные positives Не даёт модели “спамить” positives без контроля
F1-score Компромисс precision/recall Хорош, когда важны обе стороны
PR-AUC Качество precision-recall tradeoff Очень полезна при редком положительном классе
Balanced accuracy Средний recall по классам Ослабляет эффект доминирования majority
При сильном imbalance метрика должна соответствовать цене ошибки. Вопрос не “какая метрика моднее”, а “какая ошибка для бизнеса/домена реально дорогая”.

PR curve vs ROC curve при imbalance

ROC-AUC может выглядеть неплохо даже тогда, когда модель слабо справляется с minority class на практических порогах.
PR curve больше фокусируется на positive class и часто оказывается информативнее при сильном imbalance.
ROC: TPR vs FPR PR: Precision vs Recall Когда positive class редкий, PR-AUC часто отражает реальность лучше.

Практические стратегии борьбы с imbalance

1. Сначала выбрать правильную метрику задачи.
2. Сделать stratified split / stratified CV.
3. Попробовать class weights.
4. Попробовать threshold tuning.
5. При необходимости попробовать oversampling / undersampling / SMOTE.
6. Сравнить PR-AUC, recall, precision, F1 на честной валидации.
7. Всегда помнить про стоимость FP и FN в реальном контексте.

Что важно при sampling-техниках

Oversampling, undersampling и SMOTE должны применяться только внутри train-fold, а не до split / не до cross-validation.
Плохо: сделать SMOTE на всём датасете → потом split / CV Хорошо: в каждом train fold: fit resampling only there validation fold remains untouched
Иначе synthetic points и дубли могут утечь в validation, и оценка станет нечестной.

Когда oversampling и SMOTE могут вредить

1. Когда minority class очень шумный.
2. Когда minority и majority сильно пересекаются.
3. Когда synthetic points создаются в “пограничной каше”.
4. Когда у minority class очень мало реальных точек и геометрия нестабильна.
5. Когда проблема решается проще через class weights и threshold tuning.
6. Когда sampling делают до CV и получают leakage.

Какая стратегия часто работает первой

Шаг Что делать Зачем
1 Stratified split / CV Стабильная оценка minority class
2 Подобрать правильную метрику Не обманываться accuracy
3 Попробовать class_weight Простой baseline без изменения данных
4 Поиграть threshold Подогнать precision/recall под задачу
5 Проверить sampling/SMOTE Если minority по-прежнему плохо ловится

from sklearn.metrics import ( precision_score, recall_score, f1_score, average_precision_score, roc_auc_score ) pred = model.predict(X_valid) proba = model.predict_proba(X_valid)[:, 1] print("Precision:", precision_score(y_valid, pred)) print("Recall :", recall_score(y_valid, pred)) print("F1 :", f1_score(y_valid, pred)) print("PR-AUC :", average_precision_score(y_valid, proba)) print("ROC-AUC :", roc_auc_score(y_valid, proba))
from imblearn.pipeline import Pipeline from imblearn.over_sampling import SMOTE from sklearn.model_selection import StratifiedKFold, cross_val_score from sklearn.preprocessing import StandardScaler from sklearn.linear_model import LogisticRegression pipe = Pipeline([ ("scaler", StandardScaler()), ("smote", SMOTE(random_state=42)), ("clf", LogisticRegression(max_iter=1000)) ]) cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42) scores = cross_val_score( pipe, X, y, cv=cv, scoring="f1", n_jobs=-1 ) print("CV F1 scores:", scores) print("Mean F1:", scores.mean())
import numpy as np from sklearn.metrics import f1_score, precision_score, recall_score proba = model.predict_proba(X_valid)[:, 1] thresholds = np.linspace(0.05, 0.95, 19) best_thr = None best_f1 = -1 for thr in thresholds: pred_thr = (proba >= thr).astype(int) f1 = f1_score(y_valid, pred_thr) if f1 > best_f1: best_f1 = f1 best_thr = thr print("Best threshold:", best_thr) print("Best F1:", best_f1) pred_best = (proba >= best_thr).astype(int) print("Precision:", precision_score(y_valid, pred_best)) print("Recall :", recall_score(y_valid, pred_best))

Частые ошибки

Ошибка 1. Смотреть только на accuracy.
Ошибка 2. Делать random split без stratification.
Ошибка 3. Делать oversampling / SMOTE до split или до CV.
Ошибка 4. Игнорировать threshold tuning.
Ошибка 5. Выбирать метрику без понимания цены FP и FN.
Ошибка 6. Верить красивому ROC-AUC, не проверяя PR-AUC и реальные operating points.

Как объяснить imbalance на собеседовании

1. При сильном дисбалансе accuracy может быть высокой даже у плохой модели.
2. Нужно смотреть на recall, precision, F1, PR-AUC, balanced accuracy — в зависимости от цены ошибок.
3. Простые базовые техники: stratified split, class weights, threshold tuning.
4. Более продвинутые техники: oversampling, undersampling, SMOTE.
5. Все sampling-операции нужно делать только внутри train-fold, иначе получится leakage.

Итоговый takeaway

Imbalance ломает наивную интерпретацию accuracy.
Class weights — простой и сильный baseline-подход.
PR-AUC, precision, recall, F1 обычно важнее accuracy при редком positive class.
Oversampling / undersampling / SMOTE меняют обучающую выборку и могут помочь модели увидеть minority class.
Threshold tuning часто даёт большой практический эффект без смены модели.
Главный вопрос всегда один: какая ошибка в реальной задаче дороже — FP или FN?
Ментальная формула: imbalance = не “мало единиц”, а “нужна другая логика оценки, обучения и выбора порога”.
