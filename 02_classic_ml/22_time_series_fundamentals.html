2.22 Time Series Fundamentals

Time series problems break many standard ML assumptions because order matters. Random train/test split can leak the future, distribution changes over time, and seasonality can look like signal even when it is only calendar structure.

TL;DR

A time series is data indexed by time. The key ideas are trend, seasonality, stationarity, autocorrelation, lag features, rolling statistics, and time-aware validation. For ML, the main rule is: never let the future leak into training features or validation.

Why this matters

Demand forecasting, sales prediction, monitoring metrics, sensor data, finance, fraud, user activity, and model drift are time series problems. A model can look excellent with random split and fail completely in production because it accidentally learned from the future.

Core questions
  • What is the forecast horizon?
  • How frequently is data observed?
  • What information is available at prediction time?
  • Does the pattern change over time?
Most common failure
  • Using random split.
  • Computing rolling features with future rows.
  • Scaling using all data before splitting.
  • Evaluating one-step forecast while product needs 30-day forecast.

Components of a time series

y_t = trend_t + seasonality_t + residual_t
What the symbols mean
y_tobserved valueThe value of the series at time t, such as sales today.
trend_tlong-term movementSlow upward or downward direction over time.
seasonality_trepeating patternDaily, weekly, monthly, or yearly recurring structure.
residual_tunexplained partNoise and irregular effects left after trend and seasonality.
Intuition: split the signal into slow movement, repeating calendar structure, and leftover surprise.
Analogy: store sales can rise over years, spike every weekend, and still have random daily shocks.

Stationarity

A stationary series has statistical properties that do not change over time. In practice this usually means stable mean, variance, and autocorrelation structure.

E[y_t] = mu, Var(y_t) = sigma^2, Cov(y_t, y_{t-k}) depends only on k
What the symbols mean
E[y_t]meanAverage level of the series at time t.
Var(y_t)varianceHow much values fluctuate around the mean.
CovcovarianceHow values at two time points move together.
klagDistance in time steps between two observations.
Intuition: the series behaves similarly no matter where you look in time.
Analogy: a machine that vibrates around the same level all year is more stationary than a growing business metric.

Many classical methods prefer stationarity. Differencing, detrending, log transform, and seasonal adjustment are common ways to make a series easier to model.

Autocorrelation

rho_k = Corr(y_t, y_{t-k})
What the symbols mean
rho_kautocorrelation at lag kHow strongly current values relate to values k steps ago.
y_tcurrent valueThe value at current time t.
y_{t-k}lagged valueThe value k steps before current time.
CorrcorrelationScale-free linear relationship from -1 to 1.
Intuition: autocorrelation tells how much the past predicts the present.
Numeric example: if daily demand has rho_7=0.8, demand this Monday is strongly related to last Monday.

ARIMA intuition

ARIMA models combine autoregression, differencing, and moving-average error correction.

ARIMA(p,d,q) = AR(p) + differencing(d) + MA(q)
What the symbols mean
pAR orderHow many lagged values are used as predictors.
ddifferencing orderHow many times the series is differenced to reduce trend.
qMA orderHow many lagged forecast errors are modeled.
AR/MAmodel partsAR uses past values; MA uses past errors.
Intuition: predict from past values, remove trend, and correct systematic past mistakes.
Analogy: estimate tomorrow by looking at recent days, adjusting for growth, and remembering recent forecast errors.

For many production ML tasks, tree models with lag and rolling features are easier to combine with external regressors than pure ARIMA.

Feature engineering for ML

features_t = [y_{t-1}, y_{t-7}, rolling_mean(y_{t-7:t-1}), calendar_t, external_t]
What the symbols mean
y_{t-1}recent lagMost recent known value before prediction time.
y_{t-7}seasonal lagValue from the same weekday in daily data.
rolling_meanhistorical averageAverage over past values only; must not include y_t.
calendar_tcalendar featuresDay of week, month, holiday, hour, promo calendar.
external_tknown covariatesWeather forecast, price, campaign, or other known-at-time features.
Intuition: convert temporal dependence into normal tabular features while respecting time order.
Analogy: forecast demand using yesterday, last week, recent average, and known holidays.
# What it does: creates lag and rolling features without future leakage.
# Input shapes: df has columns ["date", "y"]
# Output shapes: supervised dataframe with lag features
import pandas as pd

df = pd.DataFrame({
    "date": pd.date_range("2026-01-01", periods=30, freq="D"),
    "y": [10, 11, 13, 12, 15, 18, 20, 11, 12, 14,
          13, 16, 19, 21, 12, 13, 15, 14, 17, 20,
          22, 13, 14, 16, 15, 18, 21, 23, 14, 15],
})

df = df.sort_values("date")
df["lag_1"] = df["y"].shift(1)
df["lag_7"] = df["y"].shift(7)
df["rolling_7_mean"] = df["y"].shift(1).rolling(7).mean()
df["dayofweek"] = df["date"].dt.dayofweek

features = df.dropna()
print(features.head())

# Gotchas:
# 1. Use shift(1) before rolling, otherwise y_t leaks into its own feature.
# 2. Split by time before fitting scalers, encoders, or target transforms.

Time-aware validation

train: [1..t], validation: [t+1..t+h]
What the symbols mean
tcutoff timeLast timestamp allowed in training data.
hforecast horizonHow far into the future the validation period extends.
trainpast dataOnly observations available before the cutoff.
validationfuture blockThe period used to simulate production forecasting.
Intuition: validate by pretending you are standing in the past and predicting the future.
Analogy: you cannot use next month's sales report to forecast next month.

Use rolling-origin validation when possible: train on older data, validate on the next horizon, move the cutoff forward, and repeat.

Metrics

Scale-dependent

  • MAE: robust and interpretable in original units.
  • RMSE: penalizes large errors more.
  • WAPE: useful for aggregate business error.

Scale-free

  • MAPE: intuitive but breaks near zero.
  • SMAPE: less asymmetric than MAPE.
  • MASE: compares against naive baseline.

Always compare against a naive baseline such as "tomorrow equals today" or "this Monday equals last Monday". If your model cannot beat that, feature engineering or validation is wrong.

Typical mistakes

Leakage mistakes
  • Random split across time.
  • Rolling windows that include current or future target.
  • Global normalization before time split.
  • Using features unknown at prediction time.
Modeling mistakes
  • Ignoring forecast horizon.
  • Ignoring calendar and known future covariates.
  • Trusting one validation split during regime change.
  • Using MAPE when true values are near zero.

Foundational

Find leakage

A feature is `rolling_7_mean = y.rolling(7).mean()`. Why is this wrong for one-step forecasting?

Show solution

It includes y_t in the feature for predicting y_t. Use y.shift(1).rolling(7).mean() instead.

Intermediate

Choose validation

You forecast weekly demand four weeks ahead. Design a validation split.

Show solution

Use rolling cutoffs. For each cutoff, train on past weeks and validate on the next four weeks. Average metrics across cutoffs.

Show solution
