2.7 Метрики классификации

Confusion matrix, accuracy, precision, recall, F1, ROC-AUC, PR-AUC, log loss, threshold tuning

confusion matrix accuracy precision recall F1 ROC-AUC PR-AUC threshold

Зачем вообще нужны метрики классификации

В задачах классификации модель предсказывает класс: спам / не спам, fraud / not fraud, disease / no disease, click / no click.

Но просто сказать “модель угадала” или “не угадала” почти всегда недостаточно. В разных задачах цена ошибок разная:

  • ложно пометить хорошее письмо как спам — неприятно,
  • пропустить мошенничество — может быть очень дорого,
  • не заметить болезнь — потенциально критично.
Интуиция. В классификации важно не только сколько ошибок, но и какие именно это ошибки. Именно поэтому одной accuracy почти никогда не хватает.
Самая частая ошибка новичков: смотреть только на accuracy и игнорировать imbalance, threshold и структуру ошибок.

2.7.1 Confusion matrix

Почти все базовые метрики классификации строятся из confusion matrix.

\text{Confusion Matrix} = \begin{bmatrix} TN & FP \\ FN & TP \end{bmatrix}

Positive class

Класс, который мы считаем “интересным”: болезнь, fraud, positive response, click, default и т.д.

Negative class

Обратный класс: отсутствует болезнь, нет fraude, нет клика, нет дефолта.

TN — True Negative
Модель сказала 0, и на самом деле был 0.
FP — False Positive
Модель сказала 1, но на самом деле был 0.
FN — False Negative
Модель сказала 0, но на самом деле был 1.
TP — True Positive
Модель сказала 1, и на самом деле был 1.
Confusion matrix — это фундамент. Accuracy, precision, recall, F1 и многие другие метрики — просто разные функции от TN, FP, FN, TP.

2.7.2 Accuracy

\text{Accuracy} = \frac{TP + TN}{TP + TN + FP + FN}

Accuracy показывает долю всех объектов, которые модель классифицировала правильно.

Когда accuracy удобна

  • классы примерно сбалансированы,
  • цена всех ошибок примерно одинаковая,
  • нужна простая первичная метрика.

Когда она обманывает

  • сильный class imbalance,
  • positive class редкий,
  • FN и FP имеют очень разную стоимость.
Если 99% объектов — класс 0, то модель, которая всегда предсказывает 0, даст accuracy = 99%, но практически будет бесполезной.

Почему accuracy ломается при imbalance

Представь dataset из 1000 объектов:

  • 990 — negative,
  • 10 — positive.

Модель, которая всегда говорит “negative”, получит:

\text{Accuracy} = \frac{990}{1000} = 99\%

Но recall по positive class у неё будет:

\text{Recall} = 0
Это одна из самых важных интуиций в classification: высокая accuracy может скрывать полную слепоту к редкому, но важному классу.

2.7.3 Precision / Recall

Precision

\text{Precision} = \frac{TP}{TP + FP}

Из всех объектов, которые модель назвала positive, какая доля действительно positive?

Recall

\text{Recall} = \frac{TP}{TP + FN}

Из всех реально positive объектов, какую долю модель сумела найти?

Precision отвечает на вопрос: “Если модель тревожится, насколько ей можно верить?”

Recall отвечает на вопрос: “Сколько настоящих positive случаев мы вообще не пропустили?”

Когда важнее precision, а когда recall

Сценарий Часто важнее Почему
Спам-фильтр Precision не хочется помечать хорошие письма как спам
Медицинский скрининг Recall опасно пропустить больного пациента
Fraud detection Recall или баланс пропущенный fraud может быть очень дорогим
Автоматическая модерация Баланс зависит от продукта FP и FN по-разному вредят UX
Нельзя сказать “precision лучше recall” или наоборот. Всё определяется ценой FP и FN в конкретной задаче.

2.7.4 F1-score

F1 — это гармоническое среднее precision и recall.

F1 = 2 \cdot \frac{\text{Precision} \cdot \text{Recall}} {\text{Precision} + \text{Recall}}

Можно также записать через TP, FP, FN:

F1 = \frac{2TP}{2TP + FP + FN}

Почему не обычное среднее

Гармоническое среднее сильно падает, если одна из величин маленькая. Это полезно, когда нужен именно баланс precision и recall.

Смысл F1

Высокий F1 означает, что модель и неплохо находит positives, и не слишком часто ошибается ложными тревогами.

F1 особенно популярен при imbalance, когда accuracy уже не даёт честной картины.

Почему F1 не всесилен

  • F1 вообще не учитывает TN,
  • может скрыть, что precision и recall очень разной природы,
  • не отражает quality of probability estimates,
  • не говорит, какой именно тип ошибки дороже.
Иногда две модели имеют похожий F1, но у одной высокий recall и низкий precision, а у другой наоборот. Для бизнеса это могут быть совершенно разные системы.

Откуда берутся ROC и PR curves

Многие модели классификации выдают не просто класс, а score или probability:

\hat{p}(y=1 \mid x)

Чтобы превратить probability в класс, мы выбираем threshold:

\hat{y} = \begin{cases} 1, & \hat{p}\ge t \\ 0, & \hat{p}< t \end{cases}

Если threshold менять, будут меняться TP, FP, FN, TN, а значит и precision, recall, TPR, FPR.

ROC curve и PR curve — это не “магические новые метрики”, а просто графики поведения модели при разных threshold.

2.7.5 ROC / ROC-AUC

TPR / Recall

TPR = \frac{TP}{TP + FN}

FPR

FPR = \frac{FP}{FP + TN}

ROC curve строится как график:

x = FPR,\qquad y = TPR

При разных threshold мы получаем разные пары (FPR, TPR).

ROC-AUC

Площадь под ROC-кривой. Чем ближе к 1, тем лучше модель ранжирует positives выше negatives.

Интерпретация

ROC-AUC ≈ вероятность того, что случайный positive объект получит score выше, чем случайный negative.

ROC-AUC может выглядеть слишком оптимистично при сильном class imbalance. Поэтому для редкого positive class часто важнее смотреть ещё и PR-AUC.

2.7.6 PR curve / PR-AUC

PR curve строится в координатах:

x = Recall,\qquad y = Precision

То есть это график trade-off между precision и recall при разных threshold.

Почему это важно

При сильном imbalance нас часто особенно интересует, насколько хорошо модель находит редкий positive class без лавины ложных тревог.

PR-AUC

Площадь под precision-recall curve. Часто более информативна, чем ROC-AUC, когда positive class редкий.

Практическое правило: при сильном imbalance не ограничивайся ROC-AUC — смотри PR-AUC тоже.

2.7.7 Log loss

Accuracy, precision, recall и F1 оценивают итоговые классы. Но если модель выдаёт вероятности, важно оценивать ещё и качество самих вероятностей.

\text{LogLoss} = -\frac{1}{n}\sum_{i=1}^{n} \Big( y^{(i)}\log \hat{p}^{(i)} + (1-y^{(i)})\log (1-\hat{p}^{(i)}) \Big)

Что делает log loss

  • смотрит на probability, а не только на label,
  • наказывает уверенные неправильные ответы очень сильно,
  • вознаграждает confident correct predictions.

Почему это важно

Две модели могут иметь одинаковую accuracy, но одна даёт адекватные вероятности, а другая — плохие. Log loss это различит.

Accuracy не видит разницы между probability 0.51 и 0.99, если обе дали правильный класс. Log loss — видит.

2.7.8 Threshold tuning

Очень часто лучший threshold — не 0.5.

Когда threshold можно опустить

  • важнее не пропустить positive class,
  • цена FN высокая,
  • готовы терпеть больше ложных тревог.

Когда threshold можно поднять

  • ложные тревоги очень дорогие,
  • важнее точность alarms,
  • система должна быть консервативной.
t \downarrow \Rightarrow recall \uparrow,\; precision \downarrow \text{ (обычно)}
t \uparrow \Rightarrow recall \downarrow,\; precision \uparrow \text{ (обычно)}
Threshold tuning — это bridge между ML-моделью и product/business decision. Модель даёт score, а threshold превращает его в operational policy.

Сводная таблица метрик классификации

Метрика Формула / идея Когда полезна Ограничение
Accuracy (TP+TN)/All баланс классов, одинаковая цена ошибок ломается при imbalance
Precision TP/(TP+FP) важно качество positive alarms не показывает, сколько positives пропущено
Recall TP/(TP+FN) важно не пропускать positives не показывает цену ложных тревог
F1 harmonic mean of P and R нужен баланс precision/recall не учитывает TN
ROC-AUC ранжирование positives vs negatives сравнение скоринговых моделей может быть слишком оптимистичен при imbalance
PR-AUC precision-recall tradeoff редкий positive class сильнее зависит от prevalence
Log loss NLL for probabilities важно качество вероятностей менее интуитивен, чем accuracy

Как выбирать метрику

Шаг 1: кто positive class?

Определи, какой класс реально важен в задаче.

Шаг 2: что хуже — FP или FN?

Это главный вопрос для выбора между precision и recall.

Шаг 3: нужны ли calibrated probabilities?

Если downstream-система использует probabilities, смотри log loss и calibration.

Шаг 4: есть ли imbalance?

Если да — accuracy почти всегда недостаточна.

Очень практичное правило:
  • Accuracy — только как очень грубый first look,
  • Precision / Recall / F1 — для error trade-offs,
  • ROC-AUC / PR-AUC — для score-based comparison,
  • Log loss — для вероятностных моделей.

Интерактивный калькулятор метрик классификации

Вводи истины и предсказания как 0/1 через запятую. Для probabilities — отдельное поле.

sklearn

from sklearn.metrics import ( confusion_matrix, accuracy_score, precision_score, recall_score, f1_score, log_loss ) y_true = [1,0,1,1,0,0,1,0] y_pred = [1,0,1,0,0,1,1,0] y_prob = [0.91,0.11,0.82,0.43,0.20,0.62,0.78,0.07] cm = confusion_matrix(y_true, y_pred) acc = accuracy_score(y_true, y_pred) prec = precision_score(y_true, y_pred) rec = recall_score(y_true, y_pred) f1 = f1_score(y_true, y_pred) ll = log_loss(y_true, y_prob) print("Confusion matrix:\\n", cm) print("Accuracy :", acc) print("Precision:", prec) print("Recall :", rec) print("F1 :", f1) print("LogLoss :", ll)
from sklearn.metrics import ( roc_auc_score, average_precision_score, roc_curve, precision_recall_curve ) roc_auc = roc_auc_score(y_true, y_prob) pr_auc = average_precision_score(y_true, y_prob) fpr, tpr, roc_thresholds = roc_curve(y_true, y_prob) precision, recall, pr_thresholds = precision_recall_curve(y_true, y_prob) print("ROC-AUC:", roc_auc) print("PR-AUC :", pr_auc)
import numpy as np from sklearn.metrics import precision_score, recall_score, f1_score thresholds = np.linspace(0.1, 0.9, 9) for t in thresholds: y_pred_t = (np.array(y_prob) >= t).astype(int) p = precision_score(y_true, y_pred_t, zero_division=0) r = recall_score(y_true, y_pred_t, zero_division=0) f1 = f1_score(y_true, y_pred_t, zero_division=0) print(f"t={t:.1f} | precision={p:.3f} | recall={r:.3f} | f1={f1:.3f}")

Частые ошибки

  • оценивать модель только по accuracy,
  • игнорировать class imbalance,
  • не понимать, какой класс chosen as positive,
  • сравнивать модели по labels, когда важны probabilities,
  • не настраивать threshold под задачу,
  • смотреть на F1 и не смотреть отдельно на precision и recall,
  • использовать ROC-AUC как единственную метрику при редком positive class.

Takeaway

Метрики классификации отражают разные стороны качества модели.

  • Confusion matrix — фундамент всех базовых метрик.
  • Accuracy — простая, но часто обманчивая при imbalance.
  • Precision — насколько можно доверять positive predictions.
  • Recall — сколько настоящих positives мы нашли.
  • F1 — компромисс между precision и recall.
  • ROC-AUC / PR-AUC — оценка ранжирования при разных threshold.
  • Log loss — качество самих вероятностей, а не только labels.
  • Threshold tuning — мост между моделью и реальной ценой ошибок.

Самая важная мысль: правильная метрика зависит не от привычки, а от структуры данных и цены FP/FN.
