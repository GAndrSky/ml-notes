Tree intuition, Gini, Entropy, Information Gain, regression trees, depth, pruning, overfitting

Что такое дерево решений

Decision Tree — это модель, которая принимает решение через последовательность простых правил вида: “если признак меньше порога — идём влево, иначе вправо”.

\text{if } x_j \le t \Rightarrow \text{left branch}, \qquad \text{else } \Rightarrow \text{right branch}

Повторяя такие split'ы, дерево делит пространство признаков на области, внутри которых выдаёт класс или числовой прогноз.

Интуиция. Дерево — это как цепочка вопросов. Например:

“Доход > 2000?” → “Да”
“Возраст < 25?” → “Нет”
“Есть просрочки?” → “Да”

На выходе приходит решение. Поэтому дерево очень похоже на человеческую логику типа “если..., то...”.
Одно из главных преимуществ деревьев — они naturally интерпретируемы: можно буквально прочитать путь от корня к листу как набор правил.

Основные части дерева

Root
первый split
Internal nodes
промежуточные вопросы
Leaves
финальные ответы

Root node

Самый верхний узел дерева. Здесь выбирается самый первый и часто самый важный split.

Leaf node

Конечный узел без дальнейших split'ов. В классификации лист выдаёт класс, в регрессии — числовое значение.

[Root] income > 2000? ├── No → [Leaf] class = deny └── Yes → age < 30? ├── Yes → [Leaf] class = approve └── No → debt_ratio > 0.5? ├── Yes → [Leaf] class = deny └── No → [Leaf] class = approve

2.10.1 Интуиция дерева решений

Дерево не ищет глобальную формулу вроде wᵀx + b. Вместо этого оно локально выбирает лучший вопрос в каждой точке.

Шаг 1. Смотрим на все данные в текущем узле.
Шаг 2. Перебираем кандидаты: по какому признаку и порогу разделить.
Шаг 3. Выбираем split, который сильнее всего “очищает” узел.
Шаг 4. Рекурсивно повторяем то же самое для дочерних узлов.
Ключевая идея: дерево не моделирует всё пространство сразу. Оно строит решение жадно, шаг за шагом.
Это greedy algorithm. Дерево выбирает локально лучший split сейчас, но это не гарантирует глобально оптимальное дерево целиком.

Как дерево “очищает” узлы

Если в узле смешаны разные классы, узел “грязный”. Если почти все объекты в узле одного класса, узел “чистый”.

Грязный узел

Внутри много классов вперемешку. Такой узел плохо подходит для финального предсказания.

Чистый узел

Большинство объектов одного класса. Такой узел уже даёт понятное решение.

Хороший split — это такой split, после которого дочерние узлы становятся чище, чем родительский.

2.10.2 Gini impurity

Один из самых популярных критериев impurity для classification trees — Gini impurity.

Gini = 1 - \sum_{k=1}^{K} p_k^2

Где p_k — доля класса k в узле.

Если узел полностью чистый

p_1 = 1,\; p_{others}=0 \Rightarrow Gini = 0

Если классы смешаны 50/50

Gini = 1 - (0.5^2 + 0.5^2) = 0.5
Gini можно понимать как меру неоднородности узла. Чем ближе Gini к 0, тем “чище” узел.

Интерактив: Gini impurity

Для бинарного случая: p(class 1)=p, p(class 0)=1-p. Максимальная impurity будет в центре.

2.10.3 Entropy / Information Gain

Второй классический критерий impurity — entropy.

H = -\sum_{k=1}^{K} p_k \log_2 p_k

Для бинарного случая:

H(p) = -p\log_2 p - (1-p)\log_2(1-p)

Если узел чистый

H = 0

Если узел 50/50

H = 1

Хороший split уменьшает entropy. Это уменьшение называется Information Gain:

IG = H(parent) - \sum_{children} w_j H(child_j)

Где w_j — доля объектов, попавших в соответствующий дочерний узел.

Information Gain показывает, сколько неопределённости мы убрали данным split'ом.

Gini vs Entropy

Критерий Формула Интуиция Практика
Gini 1 - Σp² неоднородность узла часто чуть быстрее
Entropy -Σp log p количество неопределённости / информации часто даёт похожие split'ы
На практике Gini и Entropy часто приводят к похожим результатам. Разница обычно не драматическая.

Как выглядит split числового признака

Для числового признака дерево ищет порог:

x_j \le t \quad \text{vs} \quad x_j > t

Алгоритм перебирает кандидаты и выбирает тот порог, который максимально уменьшает impurity.

Что важно

  • дерево умеет автоматически выбирать пороги,
  • не требует scaling как k-NN или SVM,
  • может работать и с нелинейными зависимостями через последовательность split'ов.

Но

  • дерево строит piecewise-constant модель,
  • границы получаются осе-параллельные,
  • одиночное дерево может быть нестабильным.

Интерактив: один split по порогу

Здесь есть одномерные объекты с бинарными классами. Мы двигаем порог и смотрим, насколько хорошо split “очищает” левый и правый узлы.

2.10.4 Regression trees и MSE split

Для регрессии дерево тоже работает через split'ы, но вместо impurity по классам обычно минимизирует ошибку прогноза в узлах.

\hat{y}_{leaf} = \text{mean target in leaf}

А качество split'а оценивают через суммарную квадратичную ошибку:

\text{MSE split objective} = \sum_{left}(y_i - \bar{y}_{left})^2 + \sum_{right}(y_i - \bar{y}_{right})^2

Что делает regression tree

Делит пространство так, чтобы внутри каждого листа target был как можно более однородным.

Предсказание листа

Обычно это просто среднее значение target объектов в этом листе.

Классификационное дерево ищет более чистые классы, регрессионное дерево — более однородные значения target.

Интерактив: regression tree split

Здесь split делит одномерные числовые данные. Слева и справа дерево предсказывает среднее target.

2.10.5 Depth, leaves, pruning

Чтобы дерево не росло бесконечно, его обычно ограничивают.

Параметр Смысл Эффект
max_depth максимальная глубина дерева контроль сложности
min_samples_split минимум объектов для split не даёт дробить слишком маленькие узлы
min_samples_leaf минимум объектов в листе предотвращает слишком узкие листья
max_leaf_nodes максимум листьев контроль размера дерева
ccp_alpha cost-complexity pruning обрезает лишние ветви
Глубокое дерево может почти запомнить train set. Ограничения и pruning нужны, чтобы оставить только те split'ы, которые действительно помогают generalization.

Pre-pruning vs Post-pruning

Pre-pruning

Мы заранее ограничиваем рост дерева: например, ставим max_depth или min_samples_leaf.

Post-pruning

Сначала выращиваем дерево побольше, потом обрезаем ветви, которые дают мало пользы.

\text{cost-complexity objective} = \text{tree loss} + \alpha \cdot \text{number of leaves}
Pruning — это способ явно контролировать trade-off между quality of fit и complexity of the tree.

2.10.6 Overfitting в деревьях

Одиночные деревья очень склонны к overfitting.

Почему

  • дерево может продолжать split'ить почти до полного запоминания train,
  • оно легко подстраивается под шум и выбросы,
  • локальные решения могут быть слишком специфичны для конкретной выборки.

Как это выглядит

  • train accuracy почти 100%,
  • validation accuracy заметно хуже,
  • мелкие ветви соответствуют случайным флуктуациям, а не закономерностям.
Деревья low-bias, high-variance. Это одна из причин, почему позже появятся Random Forest и Boosting: они делают деревья гораздо устойчивее.

Интерактив: глубина дерева и переобучение

Здесь показана не реальная обученная модель, а интуитивная схема: чем глубже дерево, тем лучше оно фитит train, но после некоторого момента начинает терять на validation.

Это ещё один важный bias-variance сюжет: маленькое дерево underfit'ит, слишком большое — overfit'ит.

Сильные стороны деревьев

  • очень интерпретируемы,
  • умеют ловить нелинейности,
  • не требуют обязательного scaling,
  • работают и для classification, и для regression,
  • естественно умеют работать с feature interactions через последовательность split'ов,
  • часто хорошо работают на табличных данных.
Дерево — один из самых human-readable алгоритмов classical ML.

Слабые стороны деревьев

  • склонность к overfitting,
  • нестабильность: небольшой шум в данных может сильно поменять структуру дерева,
  • axis-aligned split'ы могут быть не лучшей геометрией,
  • одиночное дерево часто слабее ансамблей из деревьев.
Именно нестабильность одиночного дерева делает bagging и Random Forest такими сильными: ансамбль усредняет хаотичность отдельных деревьев.

Короткая геометрическая интуиция

Дерево режет пространство признаков прямоугольными областями через split'ы вида x_j ≤ t.

\text{Decision regions} = \text{axis-aligned rectangles / boxes}

Плюс

Можно аппроксимировать довольно сложные формы последовательностью простых условий.

Минус

Если реальная граница диагональная или гладкая, дерево будет приближать её лесенкой.

Практические замечания

1. Не полагайся на неограниченное дерево как на production-ready baseline — оно почти наверняка overfit'ит.
2. Ограничивай глубину и minimum leaf size, особенно на маленьких данных.
3. Для tabular data дерево — отличный “bridge model” между простыми линейными моделями и будущими ансамблями.
4. Feature importance у дерева полезна, но может быть biased — позже появятся более надёжные способы интерпретации.
5. Если хочется высокой точности, очень часто next step after one tree — это Random Forest или Gradient Boosting.

Частые ошибки

  • выращивать дерево без ограничений и радоваться train score,
  • считать дерево inherently robust к шуму,
  • путать интерпретируемость дерева с гарантией хорошего generalization,
  • не сравнивать train и validation quality,
  • думать, что одиночное дерево — предел tree-based методов.

Takeaway

Decision Tree — это rule-based модель, которая рекурсивно делит данные на всё более однородные области.

  • Для классификации дерево использует impurity-критерии вроде Gini или Entropy.
  • Для регрессии дерево уменьшает разброс target внутри узлов, обычно через MSE-like objective.
  • Деревья легко интерпретируются и умеют ловить нелинейности.
  • Но одиночные деревья очень склонны к overfitting и нестабильности.
  • Поэтому ограничение глубины, размер листьев и pruning — критически важны.

Самая важная мысль: дерево решений — это не просто набор правил, а базовый строительный блок почти всех сильнейших tree-based ансамблей, которые мы будем разбирать дальше.
