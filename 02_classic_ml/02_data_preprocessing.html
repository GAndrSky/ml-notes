Зачем вообще нужна предобработка

В классическом ML качество модели очень часто определяется не только самим алгоритмом, а тем, что именно подаётся на вход и как именно это подготовлено.

Один и тот же dataset может дать:

  • слабый результат без обработки,
  • приличный результат после базового cleaning,
  • сильный результат после корректного scaling, encoding и контроля leakage.
Интуиция. Модель не понимает “смысл” признаков. Она видит только числа. Если один признак живёт в диапазоне [0,1], второй в [0,100000], третий содержит пропуски, а четвёртый — текстовые категории, то без грамотной подготовки алгоритм получает хаотичный и плохо сопоставимый сигнал.
Ключевая идея. Предобработка — это превращение сырых данных в машиночитаемое, сопоставимое, устойчивое и честно подготовленное представление, пригодное для обучения и оценки модели.
Теория глубже

Preprocessing меняет геометрию задачи

Предобработка — это не косметика перед моделью. Scaling меняет расстояния, encoding меняет пространство признаков, imputation добавляет предположения о неизвестных значениях, а feature engineering буквально создаёт новые оси, по которым модель сможет разделять данные.

Один и тот же алгоритм после preprocessing видит другую задачу. Поэтому качество preprocessing нельзя оценивать отдельно от модели и validation protocol.

  • Для k-NN и SVM масштаб признаков меняет само понятие близости.
  • Для линейных моделей encoding определяет, какие сравнения модель может выразить.
  • Для деревьев scaling часто не важен, но leakage и target encoding остаются критичными.

Что обычно входит в preprocessing pipeline

Базовые шаги

  • очистка и проверка типов данных,
  • работа с пропусками,
  • обработка категориальных признаков,
  • масштабирование числовых признаков,
  • обработка выбросов,
  • лог-преобразования и power transforms,
  • генерация признаков,
  • отбор признаков.

Критически важные правила

  • никогда не использовать test для подгонки preprocessing,
  • всё, что “учится” из данных, надо fit делать только на train,
  • validation/test проходят только через transform,
  • pipeline должен воспроизводиться одинаково на новых данных.
Raw data → cleaning → split → fit preprocessing on train → transform val/test → train model → evaluate
Что означают символы
Raw dataисходные данныеНеобработанный датасет: могут быть пропуски, выбросы, разные шкалы, строковые категории.
splitразбивка на train/val/testДелается ДО fit preprocessing — иначе параметры scaler/imputer «видят» val/test (leakage).
fit on trainобучение preprocessing на trainmean, std, min, max, percentiles — всё считается только по train. Именно эти параметры будут применяться к любым новым данным.
transform val/testприменение fitted preprocessingТе же параметры из train (μ_train, σ_train) применяются к val/test. Val/test сами ничего не «обучают» в preprocessing.
Интуиция: Preprocessing — это тоже часть модели: он должен fit на train и только transform на val/test. Иначе метрики оказываются оптимистично смещены, потому что test косвенно повлиял на preprocessing.
Аналогия: Мерная лента откалибрована по одним объектам (train) — той же лентой измеряем новые объекты (test). Калибровать ленту заново по тестовым объектам — это жульничество.
Числовой пример: Train: salary=[30K, 50K, 80K] → μ=53.3K, σ=20.8K. Test salary=200K → z=(200-53.3)/20.8≈7.0. Если бы std считался на всём dataset (включая 200K), σ стал бы больше, и z уменьшился бы — test «помог» себе в preprocessing.
Теория по месту

Предобработка как часть модели

Предобработка не является нейтральной уборкой данных. Масштабирование, заполнение пропусков, кодирование категорий и удаление выбросов меняют геометрию пространства, в котором модель ищет закономерность.

Для distance-based моделей scaling меняет расстояния, для линейных моделей меняет баланс коэффициентов, для деревьев чаще влияет меньше, но может влиять через кодирование категорий и пропуски.

Частая ошибка новичков: сначала посчитать mean/std на всём dataset, а потом делать split. Это уже утечка информации, потому что train косвенно “увидел” test.

1. Пропуски (missing values)

Пропуск — это ситуация, когда значение признака неизвестно: NaN, пустая строка, null, спец-код вроде -999 и т.д.

Почему пропуски опасны

  • многие модели просто не умеют работать с NaN,
  • пропуски могут быть неслучайными,
  • они искажают статистики и расстояния,
  • неправильный fill может скрыть полезный сигнал.

Типичные стратегии

  • drop rows / columns,
  • mean / median imputation,
  • mode для категорий,
  • constant fill вроде “Unknown”,
  • missing indicator,
  • модельная импутация.
xmissing → x̂ = mean(train)   или   x̂ = median(train)
Что означают символы
x_missingпропущенное значениеNaN, null, пустая строка или специальный код (-999, 0 вместо «нет данных») — значение, которое нужно заполнить перед подачей в модель.
x̂заполненное значение (импутация)Результат: NaN заменяется на оценку. Качество x̂ прямо влияет на то, что «увидит» модель на этом признаке.
mean(train)среднее по trainВычисляется только на train. Чувствителен к выбросам: один объект с зарплатой 10M руб исказит mean всех зарплат.
median(train)медиана по trainРобастная альтернатива: 50-й перцентиль, не реагирует на выбросы. При асимметричных распределениях (доходы, цены) обычно лучше mean.
Интуиция: Заполнение NaN — это компромисс: любой выбор (mean, median, mode, 0, модельная импутация) добавляет шум или смещение. Ключевое правило: fit imputer только на train, применять к val/test с теми же параметрами.
Аналогия: Если в ведомости не указана оценка студента, пишем среднюю оценку по группе (train). Нельзя подглядывать в оценки экзаменационного листа (test), чтобы уточнить «среднюю».
Числовой пример: age в train: [25, 30, NaN, 45, 50] → mean=37.5. Test: age=[NaN, 60] → заполняем 37.5, не пересчитываем mean по test. Если выброс 150 лет: mean=57 (сдвинут), median=37.5 (устойчив).
Если распределение признака асимметричное или с выбросами, median обычно устойчивее mean. Для категориальных признаков часто используют mode или отдельную категорию “missing”.
Ситуация Часто разумно Почему
Числовой признак, мало выбросов mean imputation просто и быстро
Числовой признак, тяжёлые хвосты median imputation устойчивость к выбросам
Категориальный признак mode / “Unknown” сохраняется дискретная природа признака
Пропуск сам по себе информативен fill + missing flag сигнал “значение отсутствует” может быть полезен
Пропуск — это не просто “дырка”. Иногда сам факт отсутствия значения связан с target. Например, “нет данных о доходе” может косвенно коррелировать с поведением клиента.
Теория глубже

Пропуск — это иногда сигнал, а не мусор

Missing values бывают разных типов. MCAR означает, что пропуск случайный и почти не несёт информации. MAR означает, что пропуск объясняется другими наблюдаемыми признаками. MNAR означает, что сам факт пропуска связан с неизвестным значением или поведением пользователя.

Если пропуски не случайны, простое среднее может стереть важный сигнал. Поэтому часто полезно добавлять indicator-признак: было значение пропущено или нет.

  • MCAR: можно использовать простую imputation, если доля пропусков мала.
  • MAR: imputation должна учитывать другие признаки.
  • MNAR: сам факт пропуска может быть предиктором target.

2. Масштабирование числовых признаков

Многие алгоритмы чувствительны к масштабу признаков: k-NN, SVM, логистическая регрессия, линейные модели с регуляризацией, PCA, градиентные методы.

Когда scaling особенно важен

  • есть расстояния или нормы,
  • есть регуляризация L1/L2,
  • признаки в разных единицах,
  • есть gradient-based optimization.

Когда менее критично

  • деревья решений,
  • random forest,
  • boosting на деревьях.

Но даже там scaling может быть полезен ради единообразного pipeline.

Standardization

z = \frac{x - \mu}{\sigma}
Что означают символы
zстандартизированное значение признакаРезультат преобразования: сколько стандартных отклонений данное значение x отстоит от среднего. После standardization: E[z]≈0, Std[z]≈1.
xисходное значение признакаНеобработанное числовое значение до масштабирования. Может иметь любой диапазон (например, зарплата в тысячах рублей).
μсреднее признака на trainВычисляется только по обучающей выборке. Центрирует признак: z = (x − μ)/σ смещает распределение так, что среднее становится 0.
σстандартное отклонение признака на trainТоже вычисляется только на train. Нормирует масштаб: после деления на σ std признака становится 1. При σ=0 делить нельзя — признак-константа удаляется.
Интуиция: Standardization переводит все признаки в единицы стандартных отклонений. Признак «возраст» в годах и «зарплата» в рублях становятся сопоставимы — модель не перекошена в сторону больших чисел.
Аналогия: Как оценки в школе: вместо «получил 87 баллов из 100» — «на 1.5 стандартных отклонения выше среднего». Позволяет сравнивать результаты из разных предметов.
Числовой пример: x=100, μ=80, σ=10. z = (100−80)/10 = 2. Значение на 2 стандартных отклонения выше среднего — довольно «высокое».

Здесь μ — mean на train, а σ — standard deviation на train. После преобразования признак обычно имеет mean около 0 и std около 1.

Min-Max scaling

x' = \frac{x - x_{min}}{x_{max} - x_{min}}
Что означают символы
x'нормализованное значение в [0,1]Результат: x' всегда в диапазоне [0,1] для значений, которые модель видела на train. На test может выйти за [0,1] при выбросах.
xисходное значение признакаНеобработанное значение до масштабирования.
x_minминимум признака на trainНижняя граница диапазона, вычисленная на обучающей выборке. Выносить на test нельзя (data leakage).
x_maxмаксимум признака на trainВерхняя граница диапазона на train. x_max − x_min — диапазон, на который нормируем.
x_max − x_minдиапазон признакаРазмах значений на train. На него делится отклонение от минимума. При x_max=x_min признак-константа — деление невозможно.
Интуиция: Min-Max scaling линейно сжимает все значения в [0,1]. Удобно для нейросетей (активации часто работают в [0,1]) и визуализации. Слабость: один выброс сжимает всё остальное к нулю.
Аналогия: Как проценты: если самый тихий звук=0dB, самый громкий=120dB, то звук в 60dB — ровно 50%. Min-Max делает то же самое с любым признаком.
Числовой пример: x=70, x_min=20, x_max=120. x' = (70−20)/(120−20) = 50/100 = 0.5. Ровно середина диапазона.

Сжимает признак в диапазон вроде [0,1]. Полезно, когда нужен контролируемый диапазон, но чувствительно к выбросам.

Robust scaling

x' = \frac{x - median}{IQR} \quad,\quad IQR = Q_{0.75} - Q_{0.25}
Что означают символы
x'масштабированный признак (robust)Результат: сколько IQR данное значение отстоит от медианы. Не ограничен в [0,1] — может быть любым числом.
xисходное значение признакаНеобработанное значение до масштабирования.
medianмедиана признака на train50-й перцентиль. Робастная замена среднего μ: на неё не влияют экстремальные выбросы.
IQRмежквартильный размахIQR = Q_0.75 − Q_0.25. Робастная замена стандартного отклонения: охватывает средние 50% данных, не реагирует на выбросы.
Q_0.75, Q_0.25третий и первый квартилиВычисляются на train. Q_0.75 — значение, ниже которого 75% данных; Q_0.25 — ниже которого 25%.
Интуиция: Robust scaling — как standardization, но вместо mean/std используются median/IQR. Выброс в 1000× не смещает медиану и не раздувает IQR, поэтому «нормальные» объекты масштабируются корректно.
Аналогия: Медиана зарплаты по городу vs. среднее: один миллиардер не обрушивает медиану, но значительно сдвигает среднее. Robust scaling «иммунен» к таким выбросам.
Числовой пример: x=200, median=100, Q_0.25=70, Q_0.75=130. IQR=60. x' = (200−100)/60 ≈ 1.67. Значение умеренно выше медианы.
Как не ошибиться

Главная опасность - leakage

Любой шаг, который учится на данных, должен fit-иться только на train. Если scaler, imputer или target encoding видит validation, метрика становится слишком оптимистичной.

Думай о preprocessing как о маленькой модели внутри большой модели. У нее тоже есть состояние, и это состояние нельзя обучать на будущих данных.

Более устойчив к выбросам, потому что опирается на квартили, а не на mean/std.

Смысл scaling: не “сделать данные красивыми”, а привести признаки к сопоставимому численному масштабу, чтобы модель не переоценивала признаки только потому, что у них большие числа.

3. Кодирование категориальных признаков

Модели обычно не понимают строковые значения вроде "red", "male", "Riga" напрямую. Категории надо перевести в числовое представление.

One-hot encoding

\text{city} \in \{Riga, Tallinn, Vilnius\} \Rightarrow [1,0,0], [0,1,0], [0,0,1]
Что означают символы
cityисходный категориальный признакСтроковая переменная {Riga, Tallinn, Vilnius}. Модели не воспринимают строки: нужно превратить их в числа.
[1,0,0]one-hot вектор для RigaБинарный вектор длиной 3 (по числу уникальных городов). Ровно один элемент = 1 (индекс Riga), остальные = 0. Нет ложного порядка: Riga ≠ 1, Tallinn ≠ 2.
→операция one-hot кодированияОдна категориальная колонка разворачивается в K бинарных признаков, где K = число уникальных значений. Для K=3: city_Riga, city_Tallinn, city_Vilnius.
Kмощность категорииЧисло уникальных значений признака. При K=1000 одна категория → 1000 новых признаков (high-cardinality проблема). Решение: target encoding, embedding.
Интуиция: One-hot устраняет ложный числовой порядок (Riga=1, Tallinn=2 — модель думала бы, что Tallinn «больше» Riga). Каждый город — независимый бинарный признак. Линейная модель учит отдельный коэффициент для каждого города.
Аналогия: Анкета с вопросом «В каком городе вы живёте?» с галочками: [✓Riga, Tallinn, Vilnius] = [1,0,0]. Галочка — не числовой балл.
Числовой пример: city∈{Riga, Tallinn, Vilnius, Helsinki} — K=4. Riga→[1,0,0,0], Helsinki→[0,0,0,1]. Для линейной модели: β_Riga·1 + β_Tallinn·0 + β_Vilnius·0 + β_Helsinki·0 = β_Riga. Модель выучит отдельный вклад каждого города.

Плюсы

  • не навязывает ложный порядок,
  • подходит для большинства линейных моделей,
  • прозрачно и интерпретируемо.

Минусы

  • раздувает размерность,
  • плохо для high-cardinality категорий,
  • редкие категории могут давать шум.

Ordinal encoding

Подходит, когда в категории реально есть порядок: low < medium < high.

Важно: нельзя кодировать номинальные категории как red=1, blue=2, green=3, если в них нет порядка. Иначе модель может решить, что green > blue > red по смыслу.

High-cardinality признаки

Для признаков вроде user_id, zip_code, product_code one-hot может быть слишком дорогим. Тогда используют:

  • frequency encoding,
  • target encoding,
  • hashing trick,
  • группировку редких категорий в “Other”.
Target encoding легко даёт leakage, если делать его на всём dataset. Его нужно строить очень аккуратно, обычно через CV-стратегии.

Интерактив: one-hot encoding

Результат encoding

Если категория не встречалась во время fit encoder, на inference надо заранее решить поведение: ignore, error, “Other”, или другую стратегию.

4. Выбросы и тяжёлые хвосты

Выбросы могут сильно влиять на средние, стандартные отклонения, линейные коэффициенты и расстояния между объектами.

Что можно делать

  • оставить как есть, если это реальные редкие случаи,
  • winsorization / clipping,
  • robust scaling,
  • лог-преобразование,
  • отдельный бинарный признак “extreme value”.

Нельзя делать автоматически

  • без понимания домена удалять все “большие” числа,
  • считать любой редкий случай ошибкой,
  • ломать полезный business signal ради “красивого графика”.

Лог-преобразование

x' = \log(1 + x)
Что означают символы
x'преобразованный признакРезультат лог-преобразования. Имеет более симметричное распределение: большие значения «сжались», маленькие изменились меньше.
xисходный признакДолжен быть ≥ 0 (зарплаты, просмотры, цены). Если x может быть отрицательным, log не применяется без дополнительного сдвига.
logнатуральный логарифмСжимает масштаб: log(1+10)≈2.4, log(1+1000)≈6.9 — разница в 100 раз по x становится разницей в ~2.9 раза по x'. Подходит для степенных распределений.
+1сдвиг для нулевых значенийПозволяет применять лог при x=0: log(1+0)=0, а не −∞. Без +1: log(0) не определён. При x>0 везде +1 часто не обязателен, но безопаснее.
Интуиция: Признаки с правосторонней асимметрией (зарплаты: большинство у нуля, редкие у 10M) плохо обрабатываются линейными моделями. log(1+x) «сжимает» хвост, делая распределение ближе к нормальному — модели легче выявить линейные закономерности.
Аналогия: Население городов: Москва 12M, малый город 50K — разница в 240 раз. После log: ln(12M)≈16.3, ln(50K)≈10.8 — разница всего в 1.5 раза. Логарифмическая шкала «выравнивает» асимметрию.
Числовой пример: salary=[1000, 5000, 10000, 500000]. log(1+x): [6.9, 8.5, 9.2, 13.1]. Std исходного признака: 218K. Std лог-признака: 2.6. Линейная модель теперь видит более равномерно распределённые значения.

Полезно для положительных признаков с сильной правой асимметрией: доход, количество просмотров, длительности, денежные суммы.

Лог “сжимает” большие значения сильнее, чем маленькие. Разница между 10 и 20 сохраняется заметной, но разница между 10000 и 10010 перестаёт доминировать.

5. Feature engineering

Иногда лучший способ улучшить модель — не брать более сложный алгоритм, а построить более умные признаки.

Сырой признак Полезная инженерия Зачем
timestamp hour, weekday, month, is_weekend извлечь циклическую структуру времени
birth_date age убрать лишнюю абсолютную дату
income, debt debt / income отношение информативнее абсолютных значений
text length log(length) убрать сильную асимметрию
city country, region, capital_flag агрегация категорий
Практическая мысль: feature engineering — это место, где соединяются математика, доменная логика и здравый смысл. Очень часто простая модель на хороших признаках сильнее сложной модели на сырых данных.

6. Train / validation / test и data leakage

Это одна из самых важных тем во всём ML-пайплайне. Можно получить “прекрасные” метрики и при этом обманывать самого себя.

\text{Leakage} = \text{любая информация из future/val/test, которая попала в train-процесс}
Что означают символы
Leakageутечка данныхСитуация, когда модель получает информацию, недоступную в реальном времени — результат метрики завышен, реальное качество хуже.
futureбудущие данныеДанные, которые в реальном deployment ещё не известны на момент предсказания (например, итог события, которого ещё не было).
val/testданные валидации и тестаЧасти датасета, отложенные для честной оценки; любая информация из них не должна проникать в train-процесс.
train-процессобучение модели и preprocessingВсё, что видит модель при обучении: fit скейлера, импьютера, энкодера — всё это должно использовать только train.
Интуиция: Leakage — это «подглядывание в ответы». Если scaler fitted на train+test, он «знает» статистику test и z-нормализация теряет смысл. Модель кажется хорошей, но на новых данных провалится.
Аналогия: Учитель случайно дал ученику листок с ответами перед контрольной — оценка «5», но знаний нет. Правильно: split сначала, fit preprocessing только на train, transform val/test теми же объектами.
Числовой пример: Признак «средняя зарплата по городу» посчитан по всему датасету включая test → scaler видит будущее. Правильно: посчитать mean только по train (например, 53 K), а test-строки трансформировать этим же числом.

Классические примеры leakage

  • scaler fitted на всём dataset,
  • imputer fitted на train+test,
  • feature построен с использованием target из будущего,
  • дубликаты одного объекта в train и test,
  • time-series split сделан случайно.

Правильная логика

  • split сначала,
  • fit preprocessing только на train,
  • transform val/test теми же объектами,
  • весь pipeline должен быть воспроизводим.
Шаг 1. Делим данные на train / val / test.
Шаг 2. На train учим imputer, encoder, scaler.
Шаг 3. Те же fitted-трансформеры применяем к val/test.
Шаг 4. Учим модель только на преобразованном train.
Шаг 5. Подбираем решения по validation.
Шаг 6. Один раз финально оцениваем на test.
Test — это не место для принятия решений по гиперпараметрам. Это финальный честный экзамен модели.
Теория глубже

Fit-state: главный источник тихого leakage

Любой preprocessing, который что-то вычисляет по данным, имеет состояние: среднее и стандартное отклонение scaler-а, частоты категорий, выбранные признаки, статистики target encoding. Это состояние должно обучаться только на train.

Validation и test имеют право только проходить через уже обученный transform. Если fit происходит на всём датасете, модель получает статистику будущих данных, даже если target напрямую не использовался.

  • fit: изучает параметры обработки на train.
  • transform: применяет уже найденные параметры к любым данным.
  • Pipeline нужен, чтобы эта дисциплина сохранялась внутри cross-validation.

Интерактив: leakage vs правильный split

Даже если difference кажется маленьким, сама идея уже сломана: train pipeline получил знание о test-распределении.

7. Почему preprocessing зависит от модели

Модель Scaling Encoding Комментарий
Linear / Logistic Regression часто важен one-hot / ordinal регуляризация чувствительна к масштабу
k-NN критичен нужны числовые признаки всё строится на расстоянии
SVM обычно очень важен нужны числовые признаки margin и kernel чувствительны к масштабу
Decision Tree обычно не нужен категории всё равно надо кодировать split по порогам, а не по расстояниям
Random Forest / GBDT редко критичен зависит от реализации обычно терпимее к сырому масштабу
PCA / k-means очень важен только числовое представление геометрия пространства меняется из-за масштаба
Вывод: preprocessing нельзя выбирать вслепую. Он должен учитывать и тип данных, и тип модели, и способ валидации.

8. sklearn Pipeline и ColumnTransformer

В production-style workflow предобработка и модель должны быть связаны в единый объект. Иначе легко получить рассинхрон между тем, как данные обрабатывались на train и на inference.

import numpy as np # train-only statistics x_train = np.array([10., 20., 30., 40., 50.]) mu = x_train.mean() sigma = x_train.std() def standardize(x, mu, sigma): return (x - mu) / sigma x_val = np.array([35., 60.]) x_train_scaled = standardize(x_train, mu, sigma) x_val_scaled = standardize(x_val, mu, sigma) print("mu =", mu) print("sigma =", sigma) print("train_scaled =", x_train_scaled) print("val_scaled =", x_val_scaled)
from sklearn.pipeline import Pipeline from sklearn.impute import SimpleImputer from sklearn.preprocessing import StandardScaler from sklearn.linear_model import LogisticRegression pipe = Pipeline([ ("imputer", SimpleImputer(strategy="median")), ("scaler", StandardScaler()), ("model", LogisticRegression(max_iter=1000)) ]) pipe.fit(X_train, y_train) y_pred = pipe.predict(X_val)
from sklearn.compose import ColumnTransformer from sklearn.pipeline import Pipeline from sklearn.impute import SimpleImputer from sklearn.preprocessing import OneHotEncoder, StandardScaler from sklearn.linear_model import LogisticRegression num_features = ["age", "income", "debt_ratio"] cat_features = ["city", "education"] num_pipe = Pipeline([ ("imputer", SimpleImputer(strategy="median")), ("scaler", StandardScaler()) ]) cat_pipe = Pipeline([ ("imputer", SimpleImputer(strategy="most_frequent")), ("onehot", OneHotEncoder(handle_unknown="ignore")) ]) preprocessor = ColumnTransformer([ ("num", num_pipe, num_features), ("cat", cat_pipe, cat_features) ]) full_pipe = Pipeline([ ("preprocess", preprocessor), ("model", LogisticRegression(max_iter=2000)) ]) full_pipe.fit(X_train, y_train) pred = full_pipe.predict(X_test)
Почему Pipeline так важен: он делает fit/transform логику структурированной, уменьшает риск leakage и позволяет одинаково применять preprocessing в train и inference.

9. Частые ошибки

  • масштабировать весь dataset до split,
  • удалять пропуски без анализа причин,
  • кодировать номинальные категории как числа с порядком,
  • забывать обработку unseen categories,
  • не сохранять fitted preprocessing для production,
  • путать cleaning и leakage-safe fitting,
  • забывать, что train и production должны проходить одинаковый pipeline,
  • считать, что trees “решают всё” и preprocessing не нужен вообще.
Хороший preprocessing — это не набор ритуалов. Это аккуратная, воспроизводимая система преобразований, согласованная с моделью и протоколом оценки.

Takeaway

Предобработка данных — это мост между сырым dataset и обучаемой моделью. Её цель не просто “почистить таблицу”, а создать корректное, устойчивое и leakage-safe представление данных.

Самое важное, что надо запомнить:

  • всё, что учится из данных, fit делается только на train,
  • масштабирование и encoding зависят от модели,
  • пропуски и выбросы надо трактовать осмысленно,
  • pipeline должен быть воспроизводим от начала до конца,
  • хороший preprocessing часто даёт не меньше пользы, чем смена алгоритма.
Теоретический слой

Preprocessing как часть модели

Предобработка определяет, какую информацию модель увидит и в каком масштабе. Пропуски, категории, даты, выбросы, редкие значения и текстовые поля меняют не только удобство данных, но и саму постановку задачи.

Самая частая ошибка — обучать scaler, imputer, encoder или feature selection на всём датасете до split-а. Тогда validation/test уже частично повлияли на train-процесс, и качество становится завышенным.

Геометрическая интуиция

  • Scaling меняет геометрию: признак в больших единицах может доминировать над расстояниями, градиентами и регуляризацией.
  • One-hot encoding превращает категории в дискретные оси без искусственного порядка.
  • Выбросы растягивают оси и могут сдвигать центр данных, поэтому robust scaling иногда меняет результат сильнее, чем модель.

Вероятностная интерпретация

  • Imputation задаёт предположение о механизме пропусков: случайные они или информативные.
  • Редкие категории похожи на низкочастотные события: объединение снижает шум, но теряет детали.
  • Distribution shift часто проявляется через новые категории, другие диапазоны и изменившуюся долю пропусков.

Оптимизационный смысл

  • Нормальный масштаб признаков делает поверхность loss более ровной для градиентных методов.
  • Encoding может резко увеличить размерность и поменять эффективную силу регуляризации.
  • Pipeline гарантирует fit только на train и transform на validation/test.

Практика и типичные ошибки

  • Раздели признаки по типам: числовые, категориальные, даты, текст, бинарные флаги, потенциальные утечки.
  • Для каждого преобразования проверь, где вызывается fit.
  • Логируй словари категорий, статистики scaling-а и доли пропусков.

Мини-пример: Если доход в рублях и возраст в годах подать в k-NN без scaling, расстояние почти полностью будет определяться доходом.

Когда применять

  • Используй тему «Предобработка данных» как практический инструмент для табличных данных, baseline-моделей и быстрых проверок гипотез.
  • Классические модели особенно полезны, когда важны скорость, интерпретируемость, небольшой датасет или строгая валидация.
  • Перед нейросетью проверь, какой результат даёт простой baseline: он задаёт нижнюю планку качества.

Типичные ошибки

  • Допускать leakage: scaling, encoding, feature selection и tuning должны обучаться только на train внутри pipeline.

  • Начни с данных: пропуски, выбросы, encoding, scaling и баланс классов часто важнее выбора алгоритма.
  • Потом меняй сложность модели: регуляризацию, глубину, число соседей, число деревьев или learning rate в зависимости от темы.
  • Фиксируй validation protocol до тюнинга, иначе улучшение может быть подгонкой под конкретный split.

Упражнения: preprocessing без leakage

Цель - научиться думать о preprocessing как о части модели, а не как о ручной чистке таблицы.

[Foundational] Где fit, где transform?

Для scaler, imputer и one-hot encoder укажи, где вызывается fit. Решение: fit только на train; validation/test получают только transform.

[Foundational] Пропуски и смысл

Для признака income с 30% missing предложи две стратегии и риск каждой. Решение: median imputation плюс missing indicator; риск - missing может быть не случайным и сам нести сигнал.

[Intermediate] Scaling и модели

Каким моделям scaling критичен, а каким почти нет? Решение: k-NN, SVM, linear/logistic и PCA чувствительны; деревья и леса обычно менее чувствительны.

[Intermediate] Код pipeline

Собери ColumnTransformer для числовых и категориальных колонок. Решение: numeric: imputer+scaler; categorical: imputer+one-hot; всё внутри Pipeline с моделью.

[Hard] Target encoding без утечки

Опиши, как сделать target encoding для категорий. Решение: encoding должен считаться out-of-fold на train и отдельно применяться к validation/test, иначе target попадёт в признаки.

Проверь себя

Вопросы для самопроверки

  1. Почему стандартизация (StandardScaler) важна для алгоритмов вроде SVM и KNN?

    KNN и SVM используют расстояния между точками. Если один признак имеет диапазон [0, 1000], а другой [0, 1], первый признак будет доминировать в расстоянии. Стандартизация (z = (x-μ)/σ) приводит все признаки к единому масштабу, устраняя эту проблему.

  2. В чём разница между imputation mean и imputation median для пропущенных значений?

    Mean imputation чувствителен к выбросам: один аномальный результат сдвигает среднее, и все заполненные значения оказываются нереалистичными. Median imputation устойчив к выбросам. Для сильно скошенных распределений медиана обычно лучше.

  3. Что такое data leakage и как его избежать?

    Data leakage — когда информация из тестовой выборки проникает в обучение, завышая оценку качества. Пример: fit StandardScaler на всём датасете до train/test split. Правило: fit любого препроцессора только на train, а transform применять к test отдельно.

  4. Когда использовать one-hot encoding, а когда ordinal encoding для категориальных признаков?

    One-hot encoding подходит для номинальных категорий (цвет, страна), где нет порядка — он не вносит ложной метрики. Ordinal encoding используют, когда порядок значим (маленький < средний < большой), или для деревьев решений, которым не важен масштаб.
