8.2 Gradient Checkpointing

Gradient checkpointing экономит память, не сохраняя часть activations во время forward. Во время backward эти activations пересчитываются. Мы платим compute, чтобы купить VRAM.

Почему важно

Полное хранение активаций требует O(L) памяти на L слоёв. Gradient checkpointing хранит только часть, пересчитывая остальные в backward — компромисс память vs вычисления.

Почему activations занимают память

Для backprop нужны промежуточные значения forward pass: входы слоев, outputs activations, attention tensors, normalization statistics. Чем глубже модель, длиннее sequence и больше batch, тем больше activation memory.

training_memory = parameters + gradients + optimizer_state + activations

У больших моделей часто bottleneck — не параметры, а activations, особенно при длинном context или больших feature maps.

Идея checkpointing

Вместо сохранения всех activations мы сохраняем только некоторые checkpoints. При backward недостающие activations пересчитываются из ближайшего checkpoint. Это сохраняет математически тот же градиент, но увеличивает время step.

memory_saved -> more_recompute -> slower_step
Интуиция: как не хранить все черновики решения, а сохранить ключевые промежуточные результаты и восстановить детали при проверке.

Когда применять

Да
Модель почти помещается, но OOM на backward; нужно увеличить sequence length или batch; compute дешевле дополнительной GPU memory.
Не первым делом
Если bottleneck уже compute или dataloader, checkpointing сделает training медленнее без пользы.

PyTorch example

# Activation checkpointing for a transformer-like block.
# Input shape: x is (batch, tokens, hidden).
# Output shape: same as x.
import torch
import torch.nn as nn
from torch.utils.checkpoint import checkpoint

class Block(nn.Module):
    def __init__(self, hidden):
        super().__init__()
        self.net = nn.Sequential(
            nn.LayerNorm(hidden),
            nn.Linear(hidden, 4 * hidden),
            nn.GELU(),
            nn.Linear(4 * hidden, hidden),
        )

    def forward(self, x):
        return x + self.net(x)

block = Block(hidden=768)
x = torch.randn(2, 1024, 768, requires_grad=True)

y = checkpoint(block, x, use_reentrant=False)
loss = y.square().mean()
loss.backward()

# Gotchas:
# 1. Inputs usually need requires_grad=True for checkpointed regions.
# 2. Random operations inside checkpointed blocks require care.

Практические trade-offs

  • Память падает, потому что меньше activations хранится между forward и backward.
  • Step time растет, потому что часть forward вычисляется повторно.
  • Debug сложнее, потому что stack traces и randomness могут быть менее очевидными.
  • Лучший эффект обычно на глубоких repeated blocks: Transformer layers, ResNet stages, U-Net blocks.

Частые ошибки

  • Checkpointить слишком мелкие операции: overhead выше пользы.
  • Ожидать ускорения. Checkpointing почти всегда замедляет step.
  • Не проверять numerical equivalence на маленьком batch.
  • Забывать, что attention memory может требовать отдельной оптимизации вроде FlashAttention.

Проверь себя

Вопросы для самопроверки

  1. Как gradient checkpointing экономит память за счёт вычислений?

    Обычно все активации каждого слоя сохраняются для backward pass — это O(depth) памяти. Gradient checkpointing сохраняет только «контрольные точки» каждые k слоёв, а промежуточные активации пересчитывает заново при backward. Компромисс: ~33% больше вычислений, но √depth меньше памяти.

  2. При каком сценарии gradient checkpointing наиболее полезен?

    При обучении очень глубоких моделей с ограниченной GPU памятью: трансформеры с 32-96 слоями на GPU 24-40GB. Также полезен при больших sequence length (долгие последовательности = большие активации). В PyTorch: torch.utils.checkpoint.checkpoint(fn, *inputs).

  3. Что такое activation offloading и чем он отличается от gradient checkpointing?

    Activation offloading перемещает активации с GPU в CPU RAM (или SSD) вместо пересчёта. Это медленнее по пропускной способности шины CPU-GPU, но не увеличивает compute. Gradient checkpointing быстрее при достаточной вычислительной мощности; offloading лучше при критическом дефиците вычислений.

  4. Как DeepSpeed реализует CPU offloading для оптимизатора?

    DeepSpeed ZeRO-Offload перемещает состояния оптимизатора (fp32 копия весов + Adam моменты) в CPU RAM. Это 3× размер модели в fp32. Обновление параметров происходит на CPU, что медленно, но позволяет обучать модели, которые иначе не помещаются в GPU. Обычно используется как крайняя мера.

Пошагово How Checkpointing Saves Memory
1 /
Шаг 01

Normal forward: all activations stored for backward — O(L) memory

In a standard forward pass PyTorch records every intermediate tensor needed by the autograd graph: layer inputs, attention scores, normalization statistics, nonlinearity outputs. For a model with L layers this means O(L) tensors sitting in VRAM simultaneously, waiting for the backward pass to consume them. A 48-layer transformer with sequence length 2048 can easily accumulate gigabytes of activation memory this way.

Шаг 02

With checkpointing: only save activations every K layers

Instead of storing every activation, checkpointing wraps segments of the network so that only the segment's input tensor (the "checkpoint") is retained. All intermediate activations within the segment are freed immediately after the forward pass through that segment. The segment boundary inputs act as waypoints from which the full computation can be reconstructed.

Шаг 03

Backward recomputes forward for each segment on demand

When the backward pass reaches a checkpointed segment, it re-runs the forward computation from the saved checkpoint input to regenerate the discarded activations. These activations are then used immediately for gradient computation and freed again. This "recompute on demand" strategy means only one segment's activations are alive in VRAM at any time during the backward pass.

Шаг 04

Trade-off: ~30% more compute, memory drops from O(L) to O(√L)

With optimal checkpoint placement every √L layers, peak activation memory drops from O(L) to O(√L) — for a 48-layer model that is roughly 7× less activation memory. The cost is that the forward pass effectively runs 1.33× times in total (once normally, plus partial recomputation during backward). In practice this translates to roughly 20–35% slower training steps, which is almost always worth it if the alternative is OOM or a smaller batch.

Проверь себя

Gradient checkpointing mechanics

  1. Why does checkpointing increase compute time?

    Because the forward pass must be run a second time, partially, during the backward pass. For each checkpointed segment, when the backward pass arrives at that segment, PyTorch re-executes the segment's forward computation from its saved input checkpoint. This regenerated activation is then used for the gradient computation and freed. The extra compute is roughly one additional full forward pass spread across all segments, adding approximately 30% wall-clock time to each training step.

  2. What is the optimal checkpoint frequency for minimum memory?

    The optimal strategy is to checkpoint every √L layers, where L is the total number of layers. This minimizes peak memory by balancing the number of stored checkpoints (O(√L)) against the number of activations that must be recomputed per segment (also O(√L)). The resulting peak activation memory is O(√L) rather than O(L) for the unmodified case. For a 36-layer transformer, checkpoint every 6 layers; for 64 layers, every 8 layers.

  3. When should you NOT use checkpointing?

    Avoid checkpointing when: (1) your model already fits comfortably in GPU memory — you pay ~30% compute overhead for no benefit; (2) training is already compute-bound, meaning the GPU is the bottleneck and adding more computation will extend run time proportionally; (3) you are using operations with randomness (e.g., dropout) inside a checkpointed segment without explicitly saving and restoring the RNG state — the recomputation will produce different random values, giving incorrect gradients. In case 3, use use_reentrant=False or manage RNG state manually.

Что ломается без этого

Using checkpointing inside a custom torch.autograd.Function that has side effects (e.g., in-place tensor modifications, writing to external buffers, mutating module state) causes those side effects to execute twice — once during the initial forward pass and once during backward recomputation. For in-place operations this corrupts the tensor values that were stored as inputs elsewhere in the graph, producing silently wrong gradients or a runtime error. Always ensure checkpointed code is side-effect-free and use use_reentrant=False for modern PyTorch.

Как закрепить тему: Gradient Checkpointing

Блок связывает теорию с практикой: где применять идею, какие ошибки проверять и как развиваться дальше.

Когда применять

  • Включай gradient checkpointing при OOM ошибках до того, как уменьшать batch size.
  • Оптимальная стратегия: checkpointing каждые √L слоёв, где L — глубина модели.
  • Комбинируй с mixed precision для максимальной экономии памяти.

Типичные ошибки

  • Включать checkpointing везде — только в узких местах по памяти, иначе лишний overhead.
  • Не измерять trade-off: checkpointing экономит память, но замедляет на ~33%.
  • Путать с activation offloading: checkpointing = recompute, offloading = CPU перемещение.

Что тюнить / проверить

  • Checkpoint granularity: каждый слой vs каждые k слоёв — баланс памяти и скорости.
  • Совмещение с torch.compile() требует осторожности — проверяй совместимость.
  • segment_size в gradient_checkpointing_enable() в Hugging Face Transformers.
