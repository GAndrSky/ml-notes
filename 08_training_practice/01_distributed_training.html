8.1 Distributed Training: DDP vs FSDP

Distributed training нужен, когда один GPU уже не дает нужный throughput или модель, контекст, batch или optimizer state не помещаются в память. Главный trade-off: compute масштабируется, но communication, память и воспроизводимость становятся сложнее.

Почему важно

Одна GPU не вмещает современные модели. Data Parallel, Tensor Parallel, Pipeline Parallel, FSDP — разные стратегии распределения, меняющие tradeoffs по памяти, скорости и коммуникации.

Что узнаешь
  • Data Parallelism — как синхронизируются градиенты через AllReduce
  • Tensor Parallelism — как матричные операции шардируются по GPU
  • Pipeline Parallelism — micro-batches и пузырь простоя
  • FSDP / ZeRO — как шардировать не только данные, но и параметры

TL;DR

DDP копирует модель на каждую GPU и синхронизирует градиенты. FSDP шардирует параметры, градиенты и optimizer state между GPU. DDP проще и часто быстрее, пока модель помещается на одну карту. FSDP нужен, когда упираемся в память.

global_batch = local_batch × number_of_workers × accumulation_steps
DDP step = forward(local batch) -> backward(local gradients) -> all_reduce(gradients) -> optimizer_step

DDP: что происходит

В Distributed Data Parallel каждый worker хранит полную копию модели. Данные делятся между workers, каждый считает gradients на своей части batch, затем gradients усредняются через all-reduce. После этого все workers делают одинаковый optimizer step и остаются синхронизированными.

Плюс
Простая mental model, хороший baseline, меньше инфраструктурных сюрпризов.
Где силен
Модель помещается на одну GPU, а нужно ускорить обучение за счет большего global batch.
Ограничение
DDP почти не уменьшает memory footprint модели на одной GPU.

FSDP: зачем шардировать

Fully Sharded Data Parallel делит параметры, gradients и optimizer state между устройствами. Перед вычислением конкретного блока нужные shard-ы собираются, после вычисления освобождаются. Это экономит память, но добавляет communication и усложняет checkpointing.

memory_per_gpu ≈ (parameters + gradients + optimizer_state) / number_of_workers + activations

На практике FSDP стоит включать не потому, что он современнее, а когда конкретная memory-проблема уже не решается mixed precision, gradient checkpointing, меньшим batch или optimizer state optimization.

Как выбирать

Начинай с DDP, если

  • модель помещается на одну GPU;
  • нужно ускорить training throughput;
  • команда только строит distributed pipeline;
  • важна простая отладка.

Переходи к FSDP, если

  • модель, optimizer state или context не помещаются;
  • activation checkpointing уже не хватает;
  • есть быстрый interconnect;
  • готовы управлять sharded checkpoints.

Минимальный DDP checklist

# DDP sanity checklist, not a full launcher.
# Input shape: each rank receives local batch (B_local, ...).
# Output shape: synchronized model parameters after optimizer.step().
import torch
from torch.nn.parallel import DistributedDataParallel as DDP

# 1. init_process_group before creating DDP wrapper
# 2. move model to local_rank
# 3. use DistributedSampler for train dataset
# 4. call sampler.set_epoch(epoch) every epoch
# 5. save checkpoints only on rank 0

model = MyModel().to(local_rank)
model = DDP(model, device_ids=[local_rank])

for epoch in range(num_epochs):
    train_sampler.set_epoch(epoch)
    for x, y in train_loader:
        x, y = x.to(local_rank), y.to(local_rank)
        loss = criterion(model(x), y)
        loss.backward()
        optimizer.step()
        optimizer.zero_grad(set_to_none=True)

# Gotchas:
# 1. Without DistributedSampler workers may see duplicated data.
# 2. Effective batch changes when world_size changes.

Частые ошибки

  • Сравнивать single-GPU и multi-GPU runs с разным effective batch без настройки learning rate.
  • Не логировать world size, global batch, accumulation steps и number of optimizer steps.
  • Считать, что больше GPU всегда ускоряют обучение линейно.
  • Игнорировать dataloader bottleneck: GPU ждут CPU/диск, а не считают.

Exercises

Foundational

Global batch

Если local batch = 8, workers = 4, accumulation = 2, чему равен effective batch?

Show solution

8 × 4 × 2 = 64. Именно этот batch влияет на optimization dynamics.

DDP vs FSDP

Модель помещается на одну GPU, но обучение долгое. Что выбрать первым?

Show solution

DDP. Он проще, хорошо масштабирует throughput и не требует sharded checkpointing.

Hard

Проверь себя

Вопросы для самопроверки

  1. Что такое Data Parallelism и как он ускоряет обучение?

    Data Parallelism: одна копия модели на каждом GPU, каждый обрабатывает разные мини-батчи. После backward pass градиенты синхронизируются (AllReduce). Effective batch size = batch_per_gpu × num_gpus. Ограничение: модель должна помещаться в памяти одного GPU.

  2. Когда нужен Model Parallelism вместо Data Parallelism?

    Когда модель не помещается в памяти одного GPU. Tensor parallelism: один слой разрезается между GPU (например, матрица W разбивается по столбцам). Pipeline parallelism: разные слои на разных GPU. LLaMA-70B не умещается даже на A100 80GB — нужен tensor/pipeline parallel.

  3. Что такое ZeRO (Zero Redundancy Optimizer) и как он экономит память?

    ZeRO (DeepSpeed) устраняет избыточность в data-parallel обучении. Stage 1: partition optimizer states. Stage 2: + partition gradients. Stage 3: + partition parameters. При Stage 3 каждый GPU хранит только 1/N параметров — что позволяет обучать модели размером N× больше при той же памяти.

  4. Что такое gradient accumulation и зачем он нужен?

    Gradient accumulation: обновляем веса каждые k мини-батчей, не каждый. Это эмулирует больший effective batch size без дополнительной памяти GPU: effective_batch = batch_size × accumulation_steps. Используется, когда нужный batch_size не влезает в память одного GPU.

Как закрепить тему: Distributed Training

Блок связывает теорию с практикой: где применять идею, какие ошибки проверять и как развиваться дальше.

Когда применять

  • DDP — первый выбор для multi-GPU на одной машине; FSDP/DeepSpeed — для мультиузловых LLM.
  • Проверяй global batch size при масштабировании: возможно нужен lr scaling.
  • Профилируй GPU utilization — часто DataLoader является bottleneck, не compute.

Типичные ошибки

  • Менять lr при изменении числа GPU без учёта linear scaling rule.
  • Игнорировать communication overhead — при слабой сети allreduce может доминировать.
  • Не проверять воспроизводимость: distributed обучение требует фиксации seed на каждом worker.

Что тюнить / проверить

  • world_size (число GPU): влияет на global batch и communication overhead.
  • gradient_as_bucket_view и find_unused_parameters в DDP.
  • backend: nccl для GPU (быстрее), gloo для CPU.
