8.4 Debugging Loss Spikes

Loss spike — это симптом, а не диагноз. Причина может быть в данных, learning rate, mixed precision, exploding gradients, scheduler, augmentation, distributed sync или баге в loss.

Почему важно

Loss spike посреди обучения — одна из самых неприятных ситуаций в ML. Понять причину — LR слишком высокий, NaN в данных, gradient explosion — означает исправить и не потерять всё обучение.

Что узнаешь
  • Как читать loss curve и отличить spike от нормального шума
  • Gradient norm мониторинг — когда clipping не помогает
  • Инспекция NaN: torch.isnan, gradient hooks, EMA loss
  • Checkpoint averaging и recovery стратегии
Нужно знать
  • Gradient clipping (4.6)
  • Numerical stability (4.9)
  • Distributed training (8.1)

Сначала классифицируй проблему

NaN / Inf
Численная авария: overflow, divide by zero, bad log, invalid target.
Sharp spike
Один или несколько batch резко ломают loss, но обучение может восстановиться.
Divergence
Loss растет систематически: часто LR, optimizer, initialization или data scale.

Базовый порядок диагностики

  1. Воспроизведи spike на минимальном subset.
  2. Залогируй batch id, input stats, target stats, loss components.
  3. Проверь `torch.isfinite` для inputs, logits, loss, gradients.
  4. Сравни run с меньшим learning rate.
  5. Отключи AMP, augmentation и distributed complexity.
  6. Проверь grad norm и включи gradient clipping как диагностику.
debugging = isolate + measure + simplify + reintroduce_complexity

PyTorch finite checks

# Finite checks for unstable training.
# Input: model, batch x/y, loss after backward.
# Output: raises error close to the real failure point.
import torch

def assert_finite(name, tensor):
    if not torch.isfinite(tensor).all():
        bad = tensor[~torch.isfinite(tensor)]
        raise RuntimeError(f"{name} has non-finite values: {bad[:5]}")

x, y = batch
assert_finite("input", x)
logits = model(x)
assert_finite("logits", logits)
loss = criterion(logits, y)
assert_finite("loss", loss)
loss.backward()

for name, param in model.named_parameters():
    if param.grad is not None:
        assert_finite(f"grad:{name}", param.grad)

grad_norm = torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
optimizer.step()

# Gotchas:
# 1. Clipping can hide a bug; use it with logging.
# 2. Check targets too: invalid labels often create impossible losses.

Где чаще всего причина

Данные
Пустые примеры, неверные labels, outliers, different scale, broken augmentation, leakage between modalities.
Optimization
Слишком высокий LR, warmup отсутствует, scheduler step не там, bad weight decay, exploding gradients.
Numerics
AMP overflow, log(0), softmax на огромных logits, divide by tiny variance, underflow probabilities.
Distributed
Разные batches на ranks, неправильный sampler, sync bugs, inconsistent loss reduction.

Что логировать постоянно

  • Train loss и отдельные loss components.
  • Learning rate и scheduler step.
  • Gradient norm до и после clipping.
  • Input mean/std/min/max по batch.
  • Logits mean/std/max и долю non-finite values.
  • Batch id или sample ids для воспроизведения.

Частые ошибки

  • Сразу менять архитектуру вместо поиска bad batch.
  • Сглаживать график и терять spike.
  • Включать clipping без логирования grad norm.
  • Не сохранять batch, на котором случился NaN.

Проверь себя

Вопросы для самопроверки

  1. Что делать, если loss вдруг резко вырос (loss spike) во время обучения?

    Алгоритм: (1) найти checkpoint до spike — сохраняет ли он состояние? (2) логировать grad norm — был ли взрыв градиентов? (3) проверить bad batch — конкретный пример с NaN/Inf? (4) снизить lr или включить gradient clipping. Часто помогает recovery: откат и пропуск плохого батча.

  2. Как определить, что модель «застряла» в плохом локальном минимуме?

    Признаки: loss перестал убывать, но grad norm ненулевой; val loss выше, чем должен быть по loss curve; loss oscillates вокруг plateau. Методы выхода: повысить lr временно (cyclical LR), добавить шум в веса, рестартовать с другой инициализацией, изменить архитектуру.

  3. Что такое dead neurons и как их обнаружить при дебаггинге?

    Dead neuron (при ReLU): активация всегда 0, градиент не проходит. Диагностика: логировать процент нулевых активаций в каждом слое. >90% нулей — проблема. Причины: слишком большой lr, плохая инициализация. Решение: LeakyReLU, меньший lr, инициализация He.

  4. Почему важно логировать не только loss, но и grad norm и learning rate?

    Loss один не даёт полной картины: может медленно убывать из-за плохого lr, несмотря на нормальные градиенты. Grad norm показывает взрывы/затухания. LR подтверждает, что schedule работает. Вместе эти три метрики позволяют быстро диагностировать большинство проблем обучения.

Rubber duck: объясни debugging loss spikes как расследование причины, а не как подбор магической настройки.

I can explain this:

Разбор ключевой формулы

θ_{t+1} = θ_t - η g_t
Что означают символы
θ_tтекущие параметрывеса модели перед update.
ηlearning rateдлина шага; слишком большой η часто и даёт spikes.
g_tградиентсигнал изменения параметров на текущем batch.
Интуиция: loss spike часто означает, что один update оказался слишком агрессивным для текущего масштаба gradients.
Числовой пример: если grad norm внезапно вырос в 50 раз, тот же learning rate может выбросить модель из стабильной области loss landscape.

Как закрепить тему: Отладка loss spikes

Блок связывает теорию с практикой: где применять идею, какие ошибки проверять и как развиваться дальше.

Когда применять

  • Всегда сохраняй checkpoint перед каждым шагом с большим grad norm.
  • Логируй grad_norm, loss, lr на каждом шаге — не только loss.
  • При NaN: первым делом проверяй входные данные, не архитектуру.

Типичные ошибки

  • Начинать с изменения архитектуры при loss spikes — сначала проверяй данные и lr.
  • Не проверять bf16/fp16 overflow как источник NaN.
  • Игнорировать накопленный bad batch — иногда достаточно его пропустить.

Что тюнить / проверить

  • Gradient clipping (max_norm): стандарт 0.5-1.0 для трансформеров.
  • LR: временное снижение помогает при спайках.
  • Loss scaling (при fp16): автоматическая стратегия в torch.cuda.amp.
