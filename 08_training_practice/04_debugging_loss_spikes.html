8.4 Debugging Loss Spikes

Loss spike — это симптом, а не диагноз. Причина может быть в данных, learning rate, mixed precision, exploding gradients, scheduler, augmentation, distributed sync или баге в loss.

Почему важно

Loss spike посреди обучения — одна из самых неприятных ситуаций в ML. Понять причину — LR слишком высокий, NaN в данных, gradient explosion — означает исправить и не потерять всё обучение.

Сначала классифицируй проблему

NaN / Inf
Численная авария: overflow, divide by zero, bad log, invalid target.
Sharp spike
Один или несколько batch резко ломают loss, но обучение может восстановиться.
Divergence
Loss растет систематически: часто LR, optimizer, initialization или data scale.

Базовый порядок диагностики

  1. Воспроизведи spike на минимальном subset.
  2. Залогируй batch id, input stats, target stats, loss components.
  3. Проверь `torch.isfinite` для inputs, logits, loss, gradients.
  4. Сравни run с меньшим learning rate.
  5. Отключи AMP, augmentation и distributed complexity.
  6. Проверь grad norm и включи gradient clipping как диагностику.
debugging = isolate + measure + simplify + reintroduce_complexity

PyTorch finite checks

# Finite checks for unstable training.
# Input: model, batch x/y, loss after backward.
# Output: raises error close to the real failure point.
import torch

def assert_finite(name, tensor):
    if not torch.isfinite(tensor).all():
        bad = tensor[~torch.isfinite(tensor)]
        raise RuntimeError(f"{name} has non-finite values: {bad[:5]}")

x, y = batch
assert_finite("input", x)
logits = model(x)
assert_finite("logits", logits)
loss = criterion(logits, y)
assert_finite("loss", loss)
loss.backward()

for name, param in model.named_parameters():
    if param.grad is not None:
        assert_finite(f"grad:{name}", param.grad)

grad_norm = torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
optimizer.step()

# Gotchas:
# 1. Clipping can hide a bug; use it with logging.
# 2. Check targets too: invalid labels often create impossible losses.

Где чаще всего причина

Данные
Пустые примеры, неверные labels, outliers, different scale, broken augmentation, leakage between modalities.
Optimization
Слишком высокий LR, warmup отсутствует, scheduler step не там, bad weight decay, exploding gradients.
Numerics
AMP overflow, log(0), softmax на огромных logits, divide by tiny variance, underflow probabilities.
Distributed
Разные batches на ranks, неправильный sampler, sync bugs, inconsistent loss reduction.

Что логировать постоянно

  • Train loss и отдельные loss components.
  • Learning rate и scheduler step.
  • Gradient norm до и после clipping.
  • Input mean/std/min/max по batch.
  • Logits mean/std/max и долю non-finite values.
  • Batch id или sample ids для воспроизведения.

Частые ошибки

  • Сразу менять архитектуру вместо поиска bad batch.
  • Сглаживать график и терять spike.
  • Включать clipping без логирования grad norm.
  • Не сохранять batch, на котором случился NaN.

Проверь себя

Вопросы для самопроверки

  1. Что делать, если loss вдруг резко вырос (loss spike) во время обучения?

    Алгоритм: (1) найти checkpoint до spike — сохраняет ли он состояние? (2) логировать grad norm — был ли взрыв градиентов? (3) проверить bad batch — конкретный пример с NaN/Inf? (4) снизить lr или включить gradient clipping. Часто помогает recovery: откат и пропуск плохого батча.

  2. Как определить, что модель «застряла» в плохом локальном минимуме?

    Признаки: loss перестал убывать, но grad norm ненулевой; val loss выше, чем должен быть по loss curve; loss oscillates вокруг plateau. Методы выхода: повысить lr временно (cyclical LR), добавить шум в веса, рестартовать с другой инициализацией, изменить архитектуру.

  3. Что такое dead neurons и как их обнаружить при дебаггинге?

    Dead neuron (при ReLU): активация всегда 0, градиент не проходит. Диагностика: логировать процент нулевых активаций в каждом слое. >90% нулей — проблема. Причины: слишком большой lr, плохая инициализация. Решение: LeakyReLU, меньший lr, инициализация He.

  4. Почему важно логировать не только loss, но и grad norm и learning rate?

    Loss один не даёт полной картины: может медленно убывать из-за плохого lr, несмотря на нормальные градиенты. Grad norm показывает взрывы/затухания. LR подтверждает, что schedule работает. Вместе эти три метрики позволяют быстро диагностировать большинство проблем обучения.

Rubber duck: объясни debugging loss spikes как расследование причины, а не как подбор магической настройки.

I can explain this:
Пошагово Loss Spike Checklist
1 /
Шаг 01

Check if the spike is in validation loss only — overfitting or data leakage

If training loss is smooth but validation loss spikes, the model is not numerically unstable — it is overfitting, or validation data is leaking from or contaminating training. Audit the preprocessing pipeline: check that no validation examples appear in the training set, that normalization statistics were computed on training data only, and that any data augmentation or tokenization is applied consistently. A val-only spike that appears suddenly mid-training often signals a data pipeline bug introduced by a recent change.

Шаг 02

Check if the spike correlates with a specific batch — log batch index, check for corrupted samples

Log the global batch index and per-sample ids whenever loss exceeds a threshold (e.g., 3× the running mean). Re-load and inspect the offending batch: check for NaN/Inf values in inputs or targets with torch.isfinite(), look for samples with extreme scale (images accidentally stored as float64, text with thousands of repeated tokens, regression targets several orders of magnitude larger than normal). A single corrupted sample in a batch can spike loss proportionally to how extreme its value is.

Шаг 03

Check gradient norms — if grad_norm > 10× usual value, you need gradient clipping

Log torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=float('inf')) before applying any clipping to see the raw gradient norm. Compare it to the baseline. If it spikes to 10–100× the normal value, the parameter update will be huge and the loss spike is a consequence. Gradient clipping (typically max_norm=1.0 for transformers) caps the update magnitude, but it treats symptoms — still investigate why the norm exploded (bad batch, too-high LR, numerics).

Шаг 04

Check learning rate schedule — cosine restart or warmup issues cause spikes at boundaries

Plot learning rate alongside loss for the same step range. Cosine annealing with warm restarts (SGDR) intentionally resets LR to a high value at cycle boundaries, which almost always causes a visible loss spike before re-convergence. A misconfigured warmup that starts too high, or a scheduler that is stepped at the wrong frequency (every epoch instead of every batch), can produce spikes that look like data problems. Verify that the scheduler's step call is placed correctly relative to optimizer.step().

Шаг 05

Check mixed precision — FP16 overflow causes NaN, verify loss scaling, switch to BF16 on Ampere+

FP16 has a maximum representable value of ~65504. If any activation or gradient exceeds this, it becomes Inf, which then propagates to NaN via Inf - Inf in subsequent ops. PyTorch's GradScaler multiplies the loss by a large scale factor before backward to prevent underflow, but if the scale is too large it causes overflow instead. Check scaler.get_scale() — if it is unusually large or dropping rapidly, that is your sign. On Ampere and newer GPUs (A100, RTX 30xx+), BF16 has the same range as FP32 and eliminates overflow issues entirely.

Проверь себя

Debugging loss spikes systematically

  1. What is the first thing to check when loss suddenly goes NaN?

    Check the inputs to the model for the batch where NaN first appeared. Use torch.isfinite(x).all() on the raw batch before it enters the model. NaN almost always enters through one of three paths: (1) a data sample with NaN/Inf values that was not cleaned, (2) FP16 overflow in the first few layers before any clipping can act, or (3) a log(0) or division by zero in the loss function (e.g., cross-entropy with a class probability of exactly zero). Finding where NaN first appears in the computational graph narrows the cause immediately.

  2. Why does gradient clipping help but not always prevent spikes?

    Gradient clipping rescales the gradient vector to have at most max_norm magnitude before the optimizer step. This prevents a single catastrophically large update from throwing the model far out of a good region. However, it does not address the root cause: if a bad batch consistently produces enormous gradients, clipping merely slows the damage. It also does not help with NaN gradients — clipping NaN values produces NaN outputs. Clipping is best thought of as a safety guardrail, not a fix. Always also log the unclipped norm to know when the guardrail is being hit.

  3. How do you confirm a data batch is causing a spike?

    Log the batch index and, if possible, sample ids whenever the loss or gradient norm exceeds a threshold. Then replay that exact batch: load the saved seed and sampler state (or store the raw batch tensor) and rerun only that step with torch.no_grad() first to see model outputs, then with full backward to see gradients. If the spike reproduces deterministically from that batch but not from adjacent batches, the batch is the cause. Also run torch.isfinite(x).all() and inspect statistical outliers: x.abs().max(), y.min(), y.max().

Что ломается без этого

Turning off gradient clipping when using large learning rates almost guarantees eventual loss explosion. Even with a well-tuned LR, any single batch with an outlier sample can produce a gradient norm that is orders of magnitude larger than usual. Without clipping, that single bad update pushes model weights far from the current optimum — sometimes irreversibly. The model may partially recover (a visible spike followed by slow recovery), or it may never recover (divergence). With clipping at max_norm=1.0, the same bad batch produces a bounded update, and training continues normally. The cost is zero when gradients are small; the benefit is a hard upper bound on the damage from any single batch.

Как закрепить тему: Отладка loss spikes

Блок связывает теорию с практикой: где применять идею, какие ошибки проверять и как развиваться дальше.

Когда применять

  • Всегда сохраняй checkpoint перед каждым шагом с большим grad norm.
  • Логируй grad_norm, loss, lr на каждом шаге — не только loss.
  • При NaN: первым делом проверяй входные данные, не архитектуру.

Типичные ошибки

  • Начинать с изменения архитектуры при loss spikes — сначала проверяй данные и lr.
  • Не проверять bf16/fp16 overflow как источник NaN.
  • Игнорировать накопленный bad batch — иногда достаточно его пропустить.

Что тюнить / проверить

  • Gradient clipping (max_norm): стандарт 0.5-1.0 для трансформеров.
  • LR: временное снижение помогает при спайках.
  • Loss scaling (при fp16): автоматическая стратегия в torch.cuda.amp.
