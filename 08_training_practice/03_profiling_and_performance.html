8.3 Profiling and Performance

Performance tuning начинается не с догадок, а с измерений. Нужно понять, где уходит время: загрузка данных, CPU preprocessing, transfer на GPU, forward, backward, optimizer или distributed communication.

Почему важно

Медленный тренировочный код = дорогое облако. Профилирование PyTorch, bottleneck analysis, MFU (Model FLOPs Utilization) — инструменты, которые позволяют выжать максимум из железа.

Главная формула step time

step_time = data_loading + host_to_device + forward + backward + optimizer + synchronization + logging

Если оптимизировать не тот компонент, training почти не ускорится. Поэтому сначала измерение, потом изменение.

PyTorch profiler

# PyTorch profiler for a training loop.
# Input: train_loader returns x, y batches.
# Output: trace files viewable in TensorBoard.
import torch
from torch.profiler import profile, ProfilerActivity, schedule, tensorboard_trace_handler

with profile(
    activities=[ProfilerActivity.CPU, ProfilerActivity.CUDA],
    schedule=schedule(wait=1, warmup=1, active=3),
    on_trace_ready=tensorboard_trace_handler("./profiler_logs"),
    record_shapes=True,
    profile_memory=True,
) as prof:
    for step, (x, y) in enumerate(train_loader):
        x, y = x.cuda(non_blocking=True), y.cuda(non_blocking=True)
        optimizer.zero_grad(set_to_none=True)
        loss = criterion(model(x), y)
        loss.backward()
        optimizer.step()
        prof.step()
        if step > 8:
            break

# Gotchas:
# 1. Skip warmup steps; first steps include one-time overhead.
# 2. Profiling itself adds overhead, so profile short windows.

Dataloader bottlenecks

Очень часто GPU простаивает не из-за модели, а из-за данных. Симптомы: sawtooth utilization, CPU на 100%, высокий time между steps, ускорение при `num_workers` tuning.

  • Увеличь `num_workers` и включи `pin_memory=True`.
  • Сократи тяжелые transforms в Python.
  • Кэшируй preprocessing, если он deterministic.
  • Проверь скорость диска и network storage.

Performance checklist

Быстрые wins
AMP/BF16, larger batch if memory allows, fused optimizer, compiled transforms, меньше logging внутри step.
Глубже
torch.compile, FlashAttention, better kernels, distributed overlap, async data prefetch.

Частые ошибки

  • Смотреть только average step time и игнорировать p95/p99.
  • Профилировать один первый batch, где много cold-start overhead.
  • Логировать изображения/большие tensors каждый step.
  • Менять несколько вещей одновременно и не понимать, что помогло.

Проверь себя

Вопросы для самопроверки

  1. Что такое GPU utilization и как его повысить?

    GPU utilization — процент времени, когда GPU активно выполняет вычисления (nvidia-smi показывает %). Низкий utilization: CPU bottleneck (медленная загрузка данных), слишком маленький batch. Повышают через num_workers в DataLoader, prefetching данных, pin_memory=True, persistent_workers=True.

  2. Что такое PyTorch Profiler и какие метрики он предоставляет?

    PyTorch Profiler (torch.profiler) измеряет время выполнения каждой операции, потребление памяти и CUDA events. Показывает: какие операции самые медленные, накладные расходы на DataLoader, overhead Python интерпретатора. TensorBoard visualizer строит flame graph и timeline.

  3. Почему torch.compile() (PyTorch 2.0) ускоряет обучение?

    torch.compile() использует TorchDynamo для захвата вычислительного графа и Triton/XLA для его компиляции. Устраняет Python overhead, применяет kernel fusion (несколько операций в одно CUDA ядро), оптимизирует memory layout. Обычно даёт 30-80% ускорение без изменения кода.

  4. Что такое memory bandwidth vs compute bottleneck и как их различить?

    Compute-bound: GPU постоянно занят вычислениями, увеличение batch размера не помогает. Memory-bound: GPU ждёт данные из памяти (много маленьких операций, attention). Диагностика: Arithmetic Intensity = FLOPs/bytes. Если AI < machine threshold — memory bound. Fusion помогает при memory bound.

Пошагово Finding Your Training Bottleneck
1 /
Шаг 01

Baseline: time one batch end-to-end

Before profiling anything specific, measure the wall-clock time for a single complete training step: data loading + host-to-device transfer + forward + backward + optimizer. Skip the first two batches to avoid cold-start overhead (CUDA initialization, JIT compilation). Record median step time over ~20 steps. This number is your baseline — everything else you measure is a fraction of it.

Шаг 02

Profile data loading — if GPU util <80% during loading, you have a data bottleneck

Watch GPU utilization with nvidia-smi dmon -s u or the profiler timeline. If utilization drops near 0% between steps in a sawtooth pattern, the GPU is idle waiting for the next batch. Fix with: increasing num_workers (start with 4–8 per GPU), setting pin_memory=True, enabling prefetch_factor=2, and caching pre-processed tensors if transforms are deterministic and expensive.

Шаг 03

Profile compute — use torch.profiler or nvidia-smi to see GPU util during forward/backward

Once data loading is ruled out, use torch.profiler with a short active window (3–5 steps after warmup) and export to TensorBoard. The CUDA timeline shows which ops dominate: attention, linear layers, normalization, or element-wise ops. High GPU util (>85%) during compute means you are compute-bound. Low util during the forward/backward itself points to Python overhead or operator serialization — consider torch.compile().

Шаг 04

Check memory bandwidth — use FLOPS utilization: actual FLOPS / theoretical peak

Model FLOPs Utilization (MFU) = (observed throughput × FLOPs per sample) / (GPU peak FLOPS). An MFU of 40–60% is typical for well-optimized transformer training. Very low MFU despite high GPU utilization indicates a memory-bandwidth bottleneck: the GPU is busy, but mostly waiting for data to move between HBM and compute cores. Memory-bound operations (e.g., layer norm, element-wise ops on large tensors) benefit from kernel fusion via FlashAttention or torch.compile().

Шаг 05

Fix the worst bottleneck first — then re-profile, do not optimize blindly

Amdahl's Law: if data loading is 60% of step time, optimizing the compute kernel by 50% only improves overall speed by ~20%. Always address the dominant bottleneck first, then re-run the full profiling cycle. Change exactly one thing between measurements so you can attribute the improvement. Common priority order: data loading → Python overhead (torch.compile) → memory bandwidth (FlashAttention, kernel fusion) → distributed communication (async all-reduce, gradient compression).

Проверь себя

Profiling and performance fundamentals

  1. What GPU utilization % indicates a data loading bottleneck?

    If GPU utilization drops below roughly 80% in a repeating sawtooth pattern between training steps, the GPU is idle waiting for the CPU to prepare the next batch. True compute bottlenecks keep the GPU at 90–100% utilization continuously. Any prolonged dip below 80% should prompt investigation of DataLoader settings: num_workers, pin_memory, prefetch_factor, and the cost of on-the-fly data transforms.

  2. What is the difference between compute-bound and memory-bandwidth-bound operations?

    Compute-bound operations spend most time executing arithmetic on data already in registers (e.g., large matrix multiplications in fully-connected layers). Adding more data to the same op does not increase time proportionally once registers are full. Memory-bandwidth-bound operations spend most time reading from or writing to HBM (e.g., softmax, layer norm, element-wise adds) — they are fast arithmetically but stall waiting for data. The distinction matters because kernel fusion (combining multiple memory-bound ops into one) is the right fix for bandwidth bottlenecks, while tensor core utilization improvements help compute-bound ones.

  3. Why does increasing batch size improve throughput up to a point?

    Larger batches amortize per-batch fixed costs (kernel launch overhead, memory allocation, data transfer latency) over more samples, increasing samples/second. GEMM (matrix multiply) efficiency also improves with larger matrices, pushing closer to peak FLOPS. However, beyond a certain batch size VRAM becomes the bottleneck — activations no longer fit, requiring gradient checkpointing or smaller batches. Additionally, very large batches may require learning rate scaling adjustments to maintain convergence quality.

Что ломается без этого

Profiling in eager mode gives misleading results for models that use torch.compile() during actual training. Compiled models fuse kernels and eliminate Python overhead, so an eager-mode profile will show bottlenecks that do not exist in production and miss bottlenecks introduced by the compilation strategy (e.g., graph breaks). Always profile with exactly the same compilation settings — same torch.compile() flags, same mode, same backend — that you use for your real training run. A profile from the wrong execution mode will send you optimizing the wrong thing.

Как закрепить тему: Профилирование и производительность

Блок связывает теорию с практикой: где применять идею, какие ошибки проверять и как развиваться дальше.

Когда применять

  • Профилируй перед оптимизацией — 80% времени часто в 20% операций.
  • Смотри на GPU utilization в nvidia-smi: если < 80% — есть bottleneck.
  • torch.compile() — простейший первый шаг оптимизации без изменения кода.

Типичные ошибки

  • Оптимизировать случайные места без профилировщика — «premature optimization».
  • Игнорировать DataLoader как bottleneck — медленная загрузка данных частая причина.
  • Сравнивать throughput без фиксации hardware, precision и batch size.

Что тюнить / проверить

  • num_workers в DataLoader: обычно 4-8 на GPU.
  • pin_memory=True и prefetch_factor — улучшают CPU→GPU transfer.
  • batch_size: больше = эффективнее использование GPU, но ограничено памятью.
