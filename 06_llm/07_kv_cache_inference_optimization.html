6.7 KV-Cache and Inference Optimization

LLM inference is usually not limited by the same things as training. During generation, memory bandwidth, KV-cache size, batching strategy, and decoding algorithm often matter more than raw FLOPs.

TL;DR

KV-cache stores keys and values from previous tokens so autoregressive decoding does not recompute the whole prefix at every new token. It saves compute, but it consumes memory proportional to layers, heads, head dimension, sequence length, and batch size.

Why this matters

In production, users care about time-to-first-token, tokens per second, total latency, throughput, and cost. A model that is easy to train can still be too expensive to serve if KV-cache memory explodes or batching is inefficient.

Key distinction
  • Prefill: process the prompt all at once.
  • Decode: generate one new token at a time.
  • KV-cache mostly helps decode, not the initial prefill pass.
Common bottlenecks
  • KV-cache memory for long contexts.
  • Memory bandwidth during token-by-token decoding.
  • Low GPU utilization with small batches.
  • Queueing latency under high traffic.

Autoregressive decoding without cache

p(x_t | x_1, ..., x_{t-1}) = Transformer(x_1, ..., x_{t-1})
What the symbols mean
x_tnext tokenThe token being generated at decode step t.
x_1...x_{t-1}prefixAll prompt and previously generated tokens.
Transformerdecoder stackAll attention and MLP layers used to compute logits for the next token.
Intuition: without caching, every new token asks the model to reread the entire conversation from the beginning.
Analogy: writing one word, then rereading the whole book before writing the next word.

This is wasteful because old tokens do not change. Their key and value projections are identical at future decode steps.

What KV-cache stores

K_l = X W_K,l, V_l = X W_V,l
What the symbols mean
K_lkeys at layer lProjected vectors used for attention matching in layer l.
V_lvalues at layer lProjected vectors that carry content mixed by attention weights.
Xhidden statesToken representations entering the attention block.
W_K,W_Vprojection weightsLearned matrices that map hidden states to key and value spaces.
Intuition: keys say what each past token offers for matching; values say what information should be copied if the token is attended to.
Analogy: store index cards once: each card has a title for lookup and content for retrieval.
KV memory ~= 2 * L * B * T * H_kv * D_head * bytes
Dimensions check
2K and VCache stores both key tensors and value tensors.
LlayersEvery transformer layer has its own attention cache.
Bbatch sizeNumber of active sequences being served together.
Tcontext lengthPrompt tokens plus generated tokens kept in cache.
H_kvKV headsMay be smaller than query heads in grouped-query attention.
D_headhead dimensionVector width per attention head.
Intuition: longer context and larger batch directly increase memory pressure.
Numeric example: L=32, B=8, T=4096, H_kv=8, D_head=128, bf16=2 bytes gives about 4 GB of KV-cache.

Prefill vs decode

Prefill

  • Processes the whole prompt in parallel.
  • Builds the initial KV-cache.
  • Usually compute-heavy for long prompts.
  • Determines time-to-first-token.

Decode

  • Generates one token per sequence step.
  • Reuses KV-cache from previous tokens.
  • Often memory-bandwidth-limited.
  • Determines tokens-per-second.
What this means in production
Long prompts hurt prefill latency and KV memory. Long outputs hurt decode time and keep cache alive longer. High throughput requires batching, but batching increases cache memory and can increase queueing latency. Serving is therefore a scheduling problem, not only a model problem.

Optimization techniques

Grouped-query attention

Use fewer KV heads than query heads. This keeps attention quality close while shrinking KV-cache memory and bandwidth.

Paged attention

Used by systems such as vLLM. KV-cache is stored in blocks like virtual memory pages, which reduces fragmentation and improves batching.

Speculative decoding

A small draft model proposes several tokens. The large model verifies them. If many are accepted, generation accelerates without changing output distribution.

Quantization

INT8 or INT4 weights reduce memory and bandwidth. Quality depends on calibration, model size, activation outliers, and whether KV-cache is also quantized.

Minimal KV-cache intuition in code

# What it does: sketches cache growth during autoregressive decoding.
# Input shapes: new_k=(B,H_kv,1,D_head), cache_k=(B,H_kv,T,D_head)
# Output shapes: updated cache_k=(B,H_kv,T+1,D_head)
import torch

B, H_kv, D_head = 2, 8, 128
cache_k = torch.empty(B, H_kv, 0, D_head)
cache_v = torch.empty(B, H_kv, 0, D_head)

for step in range(5):
    new_k = torch.randn(B, H_kv, 1, D_head)
    new_v = torch.randn(B, H_kv, 1, D_head)

    cache_k = torch.cat([cache_k, new_k], dim=2)
    cache_v = torch.cat([cache_v, new_v], dim=2)

    print("decode step", step, "cache length", cache_k.shape[2])

# Gotchas:
# 1. Real serving systems avoid repeated torch.cat because it reallocates memory.
# 2. Production caches are preallocated, paged, or managed by an inference engine.

When optimization choices matter

Small local model

llama.cpp-style quantization can matter most. The goal is to fit model weights and KV-cache in limited RAM or VRAM.

High-throughput API

vLLM-style continuous batching and paged attention can matter most. The goal is to keep GPUs utilized under many concurrent requests.

Long-context app

KV-cache dominates memory. You may need prompt compression, retrieval, sliding windows, or models with efficient attention.

Low-latency assistant

Time-to-first-token matters. Optimize prompt length, prefill, routing, model size, and queueing policy.
