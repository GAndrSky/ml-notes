6.9 RAG: Retrieval-Augmented Generation

RAG is not a magic anti-hallucination switch. It is a system that retrieves evidence, puts it into context, and asks an LLM to answer grounded in that evidence. If retrieval fails, generation usually fails too.

TL;DR

RAG combines retrieval and generation: split documents into chunks, embed them, index them, retrieve relevant chunks for a query, rerank candidates, generate an answer with citations, and evaluate both retrieval quality and answer groundedness.

Pipeline

documents -> chunks -> embeddings -> vector index -> retrieve -> rerank -> generate -> verify
What each stage does
chunkstext unitsPassages small enough to retrieve and fit into model context.
embeddingsdense vectorsNumeric representation used for semantic similarity search.
indexsearch structureFAISS, Chroma, pgvector, or another vector database / ANN index.
reranksecond-stage scoringMore expensive model reorders a smaller candidate set.
verifygrounding checkChecks whether answer claims are supported by retrieved evidence.
Intuition: retrieval gives the model a temporary memory; generation turns that memory into a useful answer.
Analogy: open the right pages of a textbook before answering a question instead of answering from memory.

Chunking

Chunking is a quality bottleneck. Chunks that are too small lose context. Chunks that are too large dilute retrieval and waste context window.

Common strategies

  • Fixed token windows with overlap.
  • Paragraph or section-based chunking.
  • Semantic chunking by topic boundaries.
  • Parent-child retrieval: retrieve small chunks, return larger context.

What to preserve

  • Document title and section hierarchy.
  • Source URL or file path.
  • Permissions and metadata.
  • Timestamp or version for freshness.
What this means in practice
Bad chunking creates impossible retrieval problems. If a chunk contains the answer but lacks the surrounding definition, generation may be vague. If a chunk is too broad, embeddings may match the wrong part of the text. Start with simple chunking, then inspect retrieved chunks manually.

Embedding similarity

cosine(q, d) = (q dot d) / (||q|| ||d||)
What the symbols mean
qquery embeddingVector representation of the user question.
ddocument embeddingVector representation of a chunk.
dotinner productMeasures alignment between two vectors.
||q||, ||d||normsNormalize vector lengths so direction matters more than magnitude.
Intuition: chunks pointing in a similar semantic direction to the query get higher scores.
Numeric example: if normalized vectors have dot product 0.82, they are much closer than a chunk with score 0.21.

Dense retrieval is strong for paraphrases. Sparse retrieval such as BM25 is strong for exact names, IDs, error codes, and rare terms. Hybrid search often beats either one alone.

Hybrid retrieval and reranking

score = alpha * dense_score + (1 - alpha) * sparse_score
What the symbols mean
dense_scoreembedding matchSemantic similarity from vector search.
sparse_scorekeyword matchLexical score from BM25 or another sparse retriever.
alphamixing weightControls how much semantic vs keyword matching matters.
Intuition: combine meaning-based search with exact-word search so neither failure mode dominates.
Analogy: find a book by both topic similarity and exact title words.

Reranking is usually a second stage: retrieve 50-200 candidates cheaply, then use a cross-encoder or LLM-based scorer on the top candidates to produce the final top-k context.

Minimal local RAG skeleton

# What it does: builds a tiny embedding search loop with cosine similarity.
# Input shapes: docs=list[str], query=str
# Output shapes: top document strings ranked by similarity
import numpy as np

docs = [
    "KV-cache stores keys and values for previous tokens during decoding.",
    "Gradient checkpointing saves memory by recomputing activations.",
    "RAG retrieves external context before generating an answer.",
]

def toy_embed(text, dim=64):
    vec = np.zeros(dim, dtype=np.float32)
    for token in text.lower().split():
        vec[hash(token) % dim] += 1.0
    norm = np.linalg.norm(vec) + 1e-12
    return vec / norm

doc_emb = np.stack([toy_embed(d) for d in docs])
query = "How does retrieval augmented generation work?"
q_emb = toy_embed(query)

scores = doc_emb @ q_emb
top_idx = scores.argsort()[::-1][:2]

for i in top_idx:
    print(float(scores[i]), docs[i])

# Gotchas:
# 1. This toy embedding is only for mechanics; use real embedding models in practice.
# 2. Always inspect retrieved chunks before blaming the generator.

Evaluation

RAG evaluation must separate retrieval and generation. If final answer is bad, first ask: did the system retrieve the needed evidence?

Retrieval metrics

  • Recall@K: did we retrieve evidence somewhere in top K?
  • MRR: how early does the first relevant chunk appear?
  • NDCG: are more relevant chunks ranked higher?
  • Context precision: how much retrieved context is useful?

Generation metrics

  • Answer correctness against reference.
  • Groundedness: claims supported by context.
  • Citation accuracy.
  • Refusal quality when evidence is missing.
Recall@K = queries_with_relevant_chunk_in_top_K / total_queries
What the symbols mean
Kretrieval cutoffNumber of chunks passed to evaluation or generation.
relevant chunkneeded evidenceA chunk that contains information required to answer the query.
total_querieseval set sizeNumber of labeled questions in the retrieval benchmark.
Intuition: before judging the LLM, check whether the answer was even available in context.
Numeric example: if 80 of 100 questions retrieve supporting evidence in top 5, Recall@5 is 0.80.

Failure modes

Retrieval failures
  • Chunk too small or too large.
  • Wrong embedding model for domain language.
  • Metadata filters remove the right document.
  • Index is stale after document updates.
Generation failures
  • LLM ignores context and answers from priors.
  • Answer cites irrelevant chunks.
  • Prompt asks for confidence the model cannot justify.
  • Context contains conflicting documents.
Production rule
Log the query, retrieved chunk IDs, scores, reranker scores, final prompt, answer, citations, latency, and user feedback. Without these logs, RAG debugging becomes guesswork.

Exercises

Foundational

Separate retrieval from generation

For five failed RAG answers, label each failure as retrieval failure, generation failure, or both.

Show solution

If the right evidence is absent from context, it is retrieval. If evidence is present but answer is wrong, it is generation. If conflicting chunks are retrieved and answer chooses poorly, it can be both.

Intermediate

Design chunking for docs

Pick chunk size and overlap for API documentation with code examples.

Show solution

Prefer section-aware chunks and keep code blocks with their explanation. Use overlap for long sections and metadata for title, endpoint, version, and permissions.

Intermediate

Choose hybrid retrieval

Explain why dense-only search may fail for error code `CUDA_ERROR_ILLEGAL_ADDRESS`.

Show solution

Exact identifiers and rare strings are often better handled by lexical search. Hybrid retrieval keeps both semantic and exact matching.

Hard

Build a small eval set

Create 30 questions with expected source chunks and compute Recall@5 manually.

Show solution

Store each query with document IDs or chunk IDs that contain the answer. Run retrieval and count how many queries include at least one gold chunk in top 5.
