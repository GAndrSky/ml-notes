ML System Design Framework

A good ML system design answer is not a model list. It is a product loop: define the objective, shape the data, train a baseline, serve predictions under constraints, monitor failures, and improve safely.

TL;DR

For interviews, use one repeatable structure: clarify the goal, define metrics, estimate scale, design data pipelines, choose model stages, describe serving, explain monitoring, and call out risks. The model is only one component of the system.

The answer framework

clarify -> metrics -> data -> baseline -> model -> serving -> monitoring -> iteration

1. Clarify the product objective

Ask what success means: engagement, revenue, latency, safety, fraud loss, user trust, or answer quality. If the objective is ambiguous, every model choice is arbitrary.

2. Define offline and online metrics

Offline metrics guide training. Online metrics decide whether the system helps users. They can disagree, so name both and explain how you would resolve conflicts.

3. Estimate scale and constraints

Number of users, items, requests per second, latency budget, storage, freshness, and cost determine whether you can use batch scoring, online inference, retrieval, or cascades.

4. Design the data loop

Specify events, labels, feature freshness, train-serving skew, negative sampling, privacy constraints, and how labels arrive. Most real ML failures are data-loop failures.

5. Start with a baseline

Use rules, logistic regression, matrix factorization, BM25, or a small CNN before heavier systems. A baseline gives a debugging anchor and a business fallback.

6. Add model stages

Large systems often use candidate generation, ranking, reranking, calibration, and guardrails. Each stage trades recall, precision, latency, and interpretability.

7. Serve under constraints

Describe online API, batch jobs, caches, feature store, vector index, model registry, rollout, fallback, and latency budget. Production is where many academic answers collapse.

8. Monitor and iterate

Track model metrics, product metrics, data drift, prediction drift, segment regressions, latency, cost, and alerting. Explain retraining triggers and rollback strategy.

Case 1 - Recommendation system

Prompt: Design a YouTube/Netflix-style recommendation system for a feed.

events -> candidate generation -> ranking -> reranking -> feed -> feedback loop

Clarify

  • Goal: maximize long-term satisfaction, not only click-through rate.
  • Users: millions of users, millions of items, fresh content every minute.
  • Constraints: low latency, diversity, safety, cold-start users and items.

Metrics

  • Offline: Recall@K, NDCG@K, MAP, calibration, coverage.
  • Online: watch time, retention, hides, skips, satisfaction surveys.
  • Guardrails: content safety, creator diversity, latency, complaints.

Data and labels

Collect impressions, clicks, watch time, skips, likes, search context, device, region, and timestamps. Correct for position bias because top-ranked items receive more exposure regardless of quality.

Model architecture

Use a two-stage system. Candidate generation retrieves hundreds or thousands of items using collaborative filtering, embeddings, ANN search, and fresh/popular rules. Ranking uses a heavier model with user, item, context, and sequence features.

Strong interview answer
Start with a simple popularity and collaborative-filtering baseline. Add candidate generation to keep latency manageable, then a ranking model for personalization. Use reranking to enforce diversity, freshness, safety, and business constraints. Monitor not only clicks but long-term retention and negative feedback, because CTR-only optimization can create clickbait.
Common mistakes
Do not propose one giant neural network scoring every item for every user online. Do not optimize only CTR. Do not ignore cold start, position bias, feedback loops, and safety constraints.

Case 2 - Real-time fraud detection

Prompt: Design a fraud detection system for payment transactions.

transaction -> feature lookup -> risk model -> threshold policy -> review / block / allow

Clarify

  • Goal: reduce fraud loss while minimizing false declines.
  • Latency: often below 100-300 ms for online decisions.
  • Labels: delayed chargebacks, manual review outcomes, confirmed fraud reports.

Metrics

  • Offline: PR-AUC, recall at fixed FPR, cost-weighted loss.
  • Online: fraud loss, false decline rate, review load, latency.
  • Segments: new users, high-value transactions, regions, merchants.

Data

Use transaction amount, merchant, device, IP, user history, velocity features, geolocation changes, failed attempts, chargeback history, and graph features. Make features point-in-time correct to avoid leakage.

Model and policy

Start with rules and gradient boosted trees. Output calibrated risk score, then use policy thresholds: allow low risk, review medium risk, block high risk. Thresholds should depend on cost and segment.

Strong interview answer
Fraud is not only classification. It is decision-making under asymmetric costs, delayed labels, adversarial behavior, and strict latency. Explain the separation between model score and business policy. The same score can lead to different actions depending on amount, user trust, region, and current fraud campaign.
Common mistakes
Do not use accuracy on imbalanced fraud data. Do not train with features that are known only after the transaction. Do not assume yesterday's fraud patterns remain stable under adversarial adaptation.

Case 3 - Semantic document search

Prompt: Design a semantic search system for internal company documents.

documents -> chunks -> embeddings -> vector index -> retrieval -> reranking -> answer / results

Clarify

  • Goal: return relevant documents or passages for natural language queries.
  • Freshness: documents may change daily or hourly.
  • Access control: users must only retrieve documents they are allowed to see.

Metrics

  • Offline: Recall@K, MRR, NDCG, labeled query-document relevance.
  • Online: click success, reformulation rate, time-to-answer.
  • Guardrails: permission leaks, stale results, duplicate chunks.

Retrieval design

Combine BM25 for exact terms with dense embeddings for semantic similarity. Use hybrid retrieval, metadata filters, chunk-level indexing, and a cross-encoder reranker for top candidates.

Serving

Precompute document embeddings, maintain an ANN index, embed each query online, retrieve candidates, filter by permissions, rerank, and return passages with document links and snippets.

Strong interview answer
Semantic search is a retrieval quality problem plus a systems problem. The important trade-off is recall versus latency: candidate generation must be broad enough to include the right answer, while reranking can be expensive because it only sees a small candidate set. Access control must be applied before returning results, not as a UI detail.
Common mistakes
Do not rely only on embeddings; exact keywords, IDs, and names matter. Do not ignore chunking strategy. Do not evaluate with a few manual queries instead of a labeled query set.

Case 4 - Image content moderation

Prompt: Design an image moderation system for user-uploaded content.

upload -> lightweight checks -> vision model -> policy layer -> human review -> feedback

Clarify

  • Goal: detect policy violations while minimizing false positives.
  • Classes: explicit content, violence, self-harm, spam, copyrighted content, minors safety.
  • Constraints: low latency for uploads, high recall for severe categories.

Metrics

  • Offline: per-class precision/recall, confusion matrix, threshold curves.
  • Online: appeal rate, moderator agreement, time-to-action.
  • Guardrails: fairness across regions, adversarial images, policy changes.

Model design

Start with a pretrained vision backbone fine-tuned for multi-label classification. Add OCR for text in images, perceptual hashing for known bad content, and human review for uncertain or high-severity cases.

Operations

Use thresholds per category. Block high-confidence severe violations, allow low-risk content, and route uncertain cases to review. Build active learning from moderator labels.

Strong interview answer
Moderation is not a single classifier. It is a policy enforcement pipeline with different thresholds for different harms. The system needs human escalation, appeal handling, auditability, and fast updates when policy changes.
Common mistakes
Do not optimize one global accuracy number. Do not ignore class-specific thresholds. Do not forget adversarial uploads, ambiguous policy boundaries, and reviewer feedback loops.

Case 5 - LLM chat assistant with RAG

Prompt: Design a company knowledge-base assistant that answers questions using internal documents.

query -> rewrite -> retrieve -> rerank -> generate with citations -> verify -> log feedback

Clarify

  • Goal: answer accurately from internal sources, with citations.
  • Constraints: privacy, access control, latency, cost, hallucination risk.
  • Output: answer, citations, uncertainty, escalation path.

Metrics

  • Retrieval: context recall, citation recall, MRR.
  • Generation: groundedness, answer correctness, refusal quality.
  • Product: resolution rate, user satisfaction, cost per answer.

Architecture

Ingest documents, chunk them, embed chunks, store vectors with metadata and permissions. At query time, rewrite ambiguous queries, retrieve, rerank, filter permissions, generate an answer with citations, and optionally verify answer support.

Safety and monitoring

Log retrieved chunks, model answer, citations, latency, feedback, and failure category. Use red-team prompts and regression sets before changing chunking, embedding model, reranker, or prompt.

Strong interview answer
RAG quality is limited by retrieval quality. If the relevant evidence is not in context, the generator must guess or refuse. A strong design separates retrieval evaluation from generation evaluation and treats citations as a product requirement, not decoration.
Common mistakes
Do not assume RAG eliminates hallucinations. Do not skip permission filtering. Do not evaluate only final answers; inspect whether the correct evidence was retrieved first.

Interview checklist

Say explicitly
  • What you optimize and what you protect as guardrails.
  • How labels are created and how delayed labels affect learning.
  • What baseline ships first and how you measure improvement.
  • Where the system can fail in production.
Avoid
  • Jumping straight to a deep model.
  • Using one metric for the whole problem.
  • Ignoring serving latency and cost.
  • Ignoring feedback loops, drift, and rollback.

Learning checkpoint

  • Can you explain the same framework without looking at the page?
  • Can you design a recommendation system with candidate generation and ranking?
  • Can you explain why fraud detection is a cost-sensitive decision system, not just classification?
  • Can you separate retrieval quality from generation quality in a RAG system?
  • Can you name monitoring signals for model, data, product, latency, and cost?
