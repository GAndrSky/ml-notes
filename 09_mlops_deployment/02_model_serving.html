9.2 Model Serving

Serving — это место, где модель становится продуктом: она получает запросы, возвращает ответы, выдерживает latency budget и не ломает downstream-системы.

Почему важно

Обученная модель бесполезна в Jupyter. REST API, GRPC, TorchServe, TensorRT — способы выставить модель в продакшн с нужными latency и throughput.

Online vs batch inference

Online inference
Запрос приходит сейчас, ответ нужен быстро. Важны p95/p99 latency, timeouts, retries, schema validation и graceful degradation.
Batch inference
Много объектов обрабатываются пачкой. Важны throughput, стоимость, idempotency, checkpoints и устойчивость к partial failure.
serving_quality = model_quality + latency + reliability + monitoring + rollback

Production contract модели

Модель в production — это не только `predict(x)`. У неё должен быть контракт: входная schema, preprocessing, output schema, version, expected ranges, error behavior и monitoring signals.

Частая ошибка: обучить модель на DataFrame с одними колонками, а в API принимать JSON без строгой проверки порядка, типов и missing values.

Минимальный FastAPI endpoint

# FastAPI serving skeleton.
# Input shape: JSON {"features": [float, float, float]}.
# Output shape: JSON {"score": float, "model_version": str}.
from fastapi import FastAPI
from pydantic import BaseModel, Field
import numpy as np

MODEL_VERSION = "linear-v1"
WEIGHTS = np.array([0.7, -0.2, 0.5], dtype=np.float32)
BIAS = -0.1

class PredictRequest(BaseModel):
    features: list[float] = Field(..., min_length=3, max_length=3)

app = FastAPI()

@app.post("/predict")
def predict(request: PredictRequest):
    x = np.array(request.features, dtype=np.float32)
    score = float(x @ WEIGHTS + BIAS)
    return {"score": score, "model_version": MODEL_VERSION}

# Run: uvicorn app:app --reload
# Gotchas:
# 1. Validate input shape and type before calling the model.
# 2. Return model_version so logs can be matched to deployments.

Latency и throughput

Latency — сколько ждёт один запрос. Throughput — сколько запросов система обрабатывает за единицу времени. Batching часто повышает throughput, но может ухудшить latency отдельного пользователя.

throughput ≈ batch_size / average_batch_time

Для real-time API обычно смотрят не среднюю latency, а p95/p99: хвост задержек определяет пользовательский опыт.

ONNX и оптимизация inference

ONNX полезен, когда модель нужно запускать вне training framework или ускорять inference через ONNX Runtime/TensorRT. Но export не бесплатен: надо проверить numerical parity, поддерживаемые ops, dynamic axes и preprocessing.

Правило: сначала сделай корректный baseline serving, потом оптимизируй runtime. Иначе можно ускорить неправильный pipeline.

Пошагово Deploying a model with FastAPI
1 /
Step 01

Load the model once at startup

Load weights, build the model, and move it to the correct device inside a startup event handler or at module level — not inside the request handler. Loading per request adds 5–10 seconds of cold-start latency on every call.

Step 02

Define input/output schema with Pydantic

Use a Pydantic BaseModel to declare the exact types, required fields, and value constraints for the request and response. FastAPI will validate and document the schema automatically, and reject malformed requests before they reach the model.

Step 03

Preprocess input exactly as in training

Apply the identical normalization, tokenization, or feature encoding that was used during training. Any mismatch — different mean/std, different tokenizer, different feature order — silently corrupts the model input and produces wrong outputs without raising an error.

Step 04

Run inference with model.eval(), no_grad(), and autocast

Call model.eval() to disable dropout and batch norm training mode. Wrap inference in torch.no_grad() to skip gradient computation and reduce memory usage. Use torch.autocast for FP16/BF16 speedup on CUDA. These three steps together halve memory and roughly double speed compared to a naive forward pass.

Step 05

Post-process and return a structured response with latency

Convert raw model output (logits, tensors) to a user-facing value (score, label, top-k items). Measure total request latency using time.perf_counter() and include it in the response or logs. Return a structured JSON object that includes the model version so logs can be matched to deployments.

Проверь себя

Вопросы для самопроверки

  1. В чём разница между batch inference и real-time (online) serving?

    Batch inference: предсказания вычисляются для большого набора примеров заранее (ночью, офлайн) — высокий throughput, задержка не критична. Online serving: предсказание по одному запросу в реальном времени — критична latency (< 100ms). Разные архитектуры, оптимизации и требования к стоимости.

  2. Что такое model quantization и как она снижает latency при serving?

    Quantization снижает точность весов (FP32 → INT8, INT4). INT8 inference в 2-4× быстрее FP32 на CPU и GPU с VNNI/CUTLASS. Размер модели также снижается в 4×. Post-training quantization (PTQ) применяется без переобучения; Quantization-Aware Training (QAT) — точнее, но требует дополнительного обучения.

  3. Что такое ONNX и зачем он нужен для deployment?

    ONNX (Open Neural Network Exchange) — открытый формат сериализации моделей, не зависящий от фреймворка. PyTorch/TF экспортируют в ONNX, который затем оптимизируется под target hardware (ONNX Runtime, TensorRT). Это даёт переносимость и обычно 2-5× ускорение inference.

  4. Что такое SLO (Service Level Objective) для ML-сервиса?

    SLO — количественные цели качества сервиса: p99 latency < 200ms, throughput > 100 RPS, availability > 99.9%. SLO определяют требования к инфраструктуре (CPU/GPU, replica count, caching). Нарушение SLO — триггер для алертинга и масштабирования.

  5. Why must preprocessing at serving match training exactly?

    The model learned weights calibrated to the exact distribution produced by training preprocessing. If serving applies different normalization values, a different tokenizer, or a different feature order, the model receives inputs it was never trained on. The output will be wrong and the error is silent — no exception is raised, just bad predictions.

  6. What is the difference between latency and throughput?

    Latency is how long one request waits for a response — measured in milliseconds, typically as p50/p95/p99. Throughput is how many requests the system handles per unit of time — measured in requests per second. A system can have high throughput (many requests processed) with high latency (each request waits a long time), which is acceptable for batch jobs but not for real-time APIs.

  7. When does batching hurt latency but help throughput?

    Batching collects multiple requests before running a single forward pass, which amortizes the GPU kernel launch overhead and increases hardware utilization — throughput rises. But each individual request now waits for the batch to fill before being processed, so p99 latency increases. Dynamic batching with a short timeout (e.g., 5ms) can balance the two, but always sets a ceiling on latency that does not exist in single-request serving.

What breaks without this

Loading the model inside the request handler means a full cold-load on every request — typically 5–10 seconds for a PyTorch model — instead of a millisecond-scale forward pass. The service appears to work but is unusable at any real traffic volume.

Как закрепить тему: Serving моделей

Блок связывает теорию с практикой: где применять идею, какие ошибки проверять и как развиваться дальше.

Когда применять

  • Начинай с простого FastAPI + Uvicorn для прототипа, переходи на TorchServe/Triton для production.
  • Измеряй p50, p95, p99 latency — не только среднюю.
  • Квантизация INT8 — часто 2-4× speedup с минимальной потерей качества.

Типичные ошибки

  • Не нагрузочное тестировать serving до production — spike трафик рушит модели.
  • Игнорировать batch inference для offline задач — это на порядок дешевле.
  • Деплоить модель без мониторинга drift — качество молча деградирует.

Что тюнить / проверить

  • Размер батча при inference: больше = выше throughput, выше latency.
  • Replicas: горизонтальное масштабирование под нагрузку.
  • Timeout и retry policy: защита от slow requests.
