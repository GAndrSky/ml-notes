9.2 Model Serving

Serving — это место, где модель становится продуктом: она получает запросы, возвращает ответы, выдерживает latency budget и не ломает downstream-системы.

Online vs batch inference

Online inference
Запрос приходит сейчас, ответ нужен быстро. Важны p95/p99 latency, timeouts, retries, schema validation и graceful degradation.
Batch inference
Много объектов обрабатываются пачкой. Важны throughput, стоимость, idempotency, checkpoints и устойчивость к partial failure.
serving_quality = model_quality + latency + reliability + monitoring + rollback

Production contract модели

Модель в production — это не только `predict(x)`. У неё должен быть контракт: входная schema, preprocessing, output schema, version, expected ranges, error behavior и monitoring signals.

Частая ошибка: обучить модель на DataFrame с одними колонками, а в API принимать JSON без строгой проверки порядка, типов и missing values.

Минимальный FastAPI endpoint

# FastAPI serving skeleton.
# Input shape: JSON {"features": [float, float, float]}.
# Output shape: JSON {"score": float, "model_version": str}.
from fastapi import FastAPI
from pydantic import BaseModel, Field
import numpy as np

MODEL_VERSION = "linear-v1"
WEIGHTS = np.array([0.7, -0.2, 0.5], dtype=np.float32)
BIAS = -0.1

class PredictRequest(BaseModel):
    features: list[float] = Field(..., min_length=3, max_length=3)

app = FastAPI()

@app.post("/predict")
def predict(request: PredictRequest):
    x = np.array(request.features, dtype=np.float32)
    score = float(x @ WEIGHTS + BIAS)
    return {"score": score, "model_version": MODEL_VERSION}

# Run: uvicorn app:app --reload
# Gotchas:
# 1. Validate input shape and type before calling the model.
# 2. Return model_version so logs can be matched to deployments.

Latency и throughput

Latency — сколько ждёт один запрос. Throughput — сколько запросов система обрабатывает за единицу времени. Batching часто повышает throughput, но может ухудшить latency отдельного пользователя.

throughput ≈ batch_size / average_batch_time

Для real-time API обычно смотрят не среднюю latency, а p95/p99: хвост задержек определяет пользовательский опыт.

ONNX и оптимизация inference

ONNX полезен, когда модель нужно запускать вне training framework или ускорять inference через ONNX Runtime/TensorRT. Но export не бесплатен: надо проверить numerical parity, поддерживаемые ops, dynamic axes и preprocessing.

Правило: сначала сделай корректный baseline serving, потом оптимизируй runtime. Иначе можно ускорить неправильный pipeline.

Разбор ключевой формулы

latency_total = preprocess + inference + postprocess + network
Что означают символы
preprocessподготовка входадекодирование, нормализация, tokenization, resize.
inferenceвремя моделичистое вычисление нейросети или ML-модели.
postprocessобработка выходаNMS, decoding, thresholding, formatting ответа.
networkсетевые задержкиserialization, transport, gateway overhead.
Интуиция: оптимизировать только model forward недостаточно — latency складывается из всей цепочки serving.
Числовой пример: 10ms preprocess + 22ms inference + 8ms postprocess + 5ms network дают 45ms total latency.
