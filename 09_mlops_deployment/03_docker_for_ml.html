9.3 Docker for ML Engineers

Docker фиксирует окружение: Python, system packages, CUDA runtime, зависимости и команду запуска. Это снижает класс ошибок “у меня работает, на сервере нет”.

Почему важно

Docker делает ML-среду воспроизводимой: один контейнер — одинаковый результат на любом железе. NVIDIA Docker позволяет прокинуть GPU внутрь контейнера.

Что узнаешь
  • Dockerfile для ML: base image, pip install, COPY — почему порядок слоёв важен
  • Docker build cache — как ускорить итерации разработки
  • nvidia-docker / --gpus all — как прокинуть GPU в контейнер
  • docker-compose для multi-service ML стеков
Нужно знать
  • Linux basics (команды)
  • Python virtual environments
  • CUDA (желательно)

Ментальная модель контейнера

Image — это слоистый read-only шаблон окружения. Container — запущенный процесс из image. Для ML это важно, потому что зависимости часто тяжелые, CUDA-sensitive и плохо воспроизводятся вручную.

image = base_os + system_deps + python_deps + source_code + entrypoint

Dockerfile для inference-сервиса

# Dockerfile
# What it does: builds a minimal FastAPI inference image.
# Input: FastAPI app in app.py and requirements.txt.
# Output: reproducible image exposing port 8000.
# Input shapes: N/A for Docker; runtime request/response schema is defined by app.py.
# Output shapes: N/A for Docker; container interface is HTTP on port 8000.
FROM python:3.11-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY app.py .

EXPOSE 8000
CMD ["uvicorn", "app:app", "--host", "0.0.0.0", "--port", "8000"]

# Gotchas:
# 1. Copy requirements before source code to reuse Docker layer cache.
# 2. Do not bake secrets or API keys into the image.

GPU containers

Для GPU важна совместимость: host driver, CUDA runtime внутри контейнера, PyTorch build и NVIDIA Container Toolkit. Контейнер не приносит драйвер GPU сам по себе: он использует host driver через runtime.

Если PyTorch видит CPU вместо GPU, проверяй не только код, но и `--gpus all`, NVIDIA runtime, версию driver и CUDA compatibility.

Что делает Docker хорошим для ML

Reproducibility
Одинаковое окружение между локальной машиной, CI и сервером.
Isolation
Проекты не ломают зависимости друг друга.
Deployment
Один artifact можно запускать в Kubernetes, VM или managed serving.
Cost
Большие images, медленные builds и cache misses могут тормозить workflow.

Практический checklist

  • Фиксируй версии в `requirements.txt` или lock-файле.
  • Используй `.dockerignore`, чтобы не копировать datasets, checkpoints и `.git`.
  • Разделяй training image и inference image, если training dependencies тяжелые.
  • Проверяй startup command локально через `docker run`, а не только через compose.

Проверь себя

Вопросы для самопроверки

  1. Почему Docker полезен для ML-проектов?

    Docker гарантирует воспроизводимость: одинаковое окружение на laptop, CI/CD и production. Решает проблему «у меня работает»: CUDA версия, Python зависимости, системные библиотеки — всё заморожено в image. Упрощает deployment: один docker run вместо установки десятков пакетов.

  2. Как правильно организовать Dockerfile для ML-проекта?

    Base image: nvidia/cuda или python:3.x. Затем: установка системных зависимостей, COPY requirements.txt + pip install (перед кодом — для кеширования слоёв), COPY код. Модель лучше загружать при запуске, а не в image (большой размер). Multi-stage build снижает финальный размер.

  3. Что такое layer caching в Docker и как его использовать?

    Docker кеширует каждый слой образа. Если COPY requirements.txt + pip install идут перед COPY кода, изменение кода не инвалидирует кеш зависимостей. Это ускоряет rebuild. Правило: изменяющиеся редко слои — раньше; изменяющиеся часто (код) — позже.

  4. Как передавать GPU в Docker контейнер?

    Для NVIDIA GPU: docker run --gpus all ... (требует nvidia-container-toolkit). В Docker Compose: deploy.resources.reservations.devices с capabilities: [gpu]. Проверка внутри контейнера: nvidia-smi. Без флага --gpus GPU не виден контейнеру.

Разбор ключевой формулы

reproducible image = base image + pinned deps + code + runtime config
Что означают символы
base imageсреда ОСстартовый слой контейнера с Python/CUDA/system libs.
pinned depsфиксированные зависимоститочные версии пакетов, без которых поведение может плавать.
codeприложениескрипты обучения, serving или inference.
runtime configконфиг запускаenv vars, entrypoint и параметры окружения.
Интуиция: Docker нужен, чтобы один и тот же код запускался в предсказуемой среде.

Как закрепить тему: Docker для ML

Блок связывает теорию с практикой: где применять идею, какие ошибки проверять и как развиваться дальше.

Когда применять

  • Используй official CUDA base images — они проверены на совместимость.
  • Multi-stage builds: builder stage (pip install) + final stage (только нужное) — меньший образ.
  • docker-compose для локальной разработки со всеми зависимостями.

Типичные ошибки

  • Хранить большие файлы (веса модели) в Docker image — используй volume mount.
  • Не тестировать GPU-доступность в контейнере (nvidia-smi) перед запуском.
  • Игнорировать .dockerignore — случайно добавляешь data/ и .venv/ в image.

Что тюнить / проверить

  • Base image: nvidia/cuda:12.x-cudnn-devel vs runtime — devel нужен только для компиляции.
  • Порт (--port) и переменные окружения через -e или .env файл.
  • Memory limit (--memory) — важен для предотвращения OOM на shared серверах.
