9.4 ML System Design Patterns

ML system design — это не “какую модель выбрать”, а как построить систему вокруг модели: данные, признаки, обучение, serving, monitoring, rollback и бизнес-метрики.

Базовая схема production ML

raw data -> validation -> features -> training -> registry -> serving -> monitoring -> retraining

Любой production ML system design можно разложить на эти этапы. Отличается только latency, объем данных, требования к качеству, стоимость ошибки и способ обновления модели.

Online vs offline inference

Offline scoring
Считаем predictions заранее: рекомендации на день, risk scores, batch moderation. Дешевле и проще, но менее свежо.
Online scoring
Считаем prediction в момент запроса. Свежее и персональнее, но требует latency budget, fallback и надежный serving.

Feature store

Feature store решает проблему training-serving skew: признаки для обучения и для online inference должны считаться одинаково. Он хранит definitions, offline values, online low-latency access и metadata.

Если training pipeline и serving pipeline считают признаки разным кодом, модель может быть хорошей offline и плохой online.

Versioning и rollback

Версионировать нужно не только модель. Нужны версии данных, feature definitions, preprocessing, threshold, calibration, tokenizer и model artifact. Rollback должен возвращать совместимый набор, а не только старый checkpoint.

deployment_version = model + features + preprocessing + threshold + config

Monitoring: что смотреть

System
Latency, error rate, saturation, memory, CPU/GPU utilization.
Data
Schema drift, missing values, distribution shift, outliers, feature freshness.
Model
Prediction drift, calibration, delayed labels, segment quality, business KPI.

A/B testing моделей

Новая модель может улучшать offline metric и ухудшать бизнес. A/B test проверяет causal impact на реальных пользователях. Важно заранее выбрать primary metric, guardrail metrics и ramp-up plan.

Guardrail examples: latency не выше X, complaint rate не растет, conversion не падает, fairness segment не деградирует.

Exercises

Foundational

Pipeline decomposition

Разложи fraud detection system на data, features, model, serving, monitoring.

Show solution

Events/transactions, velocity features, supervised model, online low-latency scoring, delayed label monitoring and drift alerts.

Intermediate

Drift strategy

Какие drift signals стоит мониторить для recommender system?

Show solution

User distribution, item popularity, missing features, CTR by segment, prediction distribution, exploration traffic, feedback delay.

Hard

Rollback design

Опиши rollback для модели, где менялись признаки и threshold.

Show solution

Rollback должен переключить model artifact, feature definitions, preprocessing version, threshold, calibration and config together.
