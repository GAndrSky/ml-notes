9.4 ML System Design Patterns

ML system design — это не “какую модель выбрать”, а как построить систему вокруг модели: данные, признаки, обучение, serving, monitoring, rollback и бизнес-метрики.

Почему важно

ML в продакшне — это не только модель. Feature Store, Shadow Mode, A/B testing, Canary deployment — паттерны, которые отделяют proof of concept от надёжного сервиса.

Что узнаешь
  • Feature Store — зачем хранить признаки отдельно от модели
  • Shadow mode — как тестировать новую модель без риска для пользователей
  • Online vs Offline inference — компромиссы и архитектурные паттерны
  • Model drift monitoring — как детектировать деградацию качества

Базовая схема production ML

raw data -> validation -> features -> training -> registry -> serving -> monitoring -> retraining

Любой production ML system design можно разложить на эти этапы. Отличается только latency, объем данных, требования к качеству, стоимость ошибки и способ обновления модели.

Online vs offline inference

Offline scoring
Считаем predictions заранее: рекомендации на день, risk scores, batch moderation. Дешевле и проще, но менее свежо.
Online scoring
Считаем prediction в момент запроса. Свежее и персональнее, но требует latency budget, fallback и надежный serving.

Feature store

Feature store решает проблему training-serving skew: признаки для обучения и для online inference должны считаться одинаково. Он хранит definitions, offline values, online low-latency access и metadata.

Если training pipeline и serving pipeline считают признаки разным кодом, модель может быть хорошей offline и плохой online.

Versioning и rollback

Версионировать нужно не только модель. Нужны версии данных, feature definitions, preprocessing, threshold, calibration, tokenizer и model artifact. Rollback должен возвращать совместимый набор, а не только старый checkpoint.

deployment_version = model + features + preprocessing + threshold + config

Monitoring: что смотреть

System
Latency, error rate, saturation, memory, CPU/GPU utilization.
Data
Schema drift, missing values, distribution shift, outliers, feature freshness.
Model
Prediction drift, calibration, delayed labels, segment quality, business KPI.

A/B testing моделей

Новая модель может улучшать offline metric и ухудшать бизнес. A/B test проверяет causal impact на реальных пользователях. Важно заранее выбрать primary metric, guardrail metrics и ramp-up plan.

Guardrail examples: latency не выше X, complaint rate не растет, conversion не падает, fairness segment не деградирует.

Пошагово Designing a prediction service
1 /
Step 01

Define SLOs

Start by agreeing on Service Level Objectives: p99 latency target (e.g., 200ms), minimum throughput (e.g., 500 RPS), and availability percentage. SLOs drive every subsequent architectural decision — model size, serving hardware, batching strategy, and replica count.

Step 02

Choose online vs batch inference

If the latency requirement is below a few hundred milliseconds and predictions must reflect real-time context, use online inference. If freshness can be measured in hours or days (e.g., daily recommendations, risk scores), batch inference is cheaper and simpler. Most systems use both: batch for precomputed defaults, online for personalized overrides.

Step 03

Design the feature pipeline

Separate features into two groups: real-time features derived from the incoming request (user query, item clicked, current timestamp) and precomputed features retrieved from a feature store (user profile embeddings, item statistics). Using a feature store ensures training and serving compute features the same way, preventing train-serve skew.

Step 04

Add monitoring

Instrument four layers: system health (latency, error rate, CPU/GPU), data drift (input feature distribution vs training baseline), prediction drift (output score distribution shift), and model performance vs business metrics (click rate, revenue, complaint rate) once delayed labels arrive. Alert on PSI > 0.2 for drift and p99 latency > SLO.

Проверь себя

Вопросы для самопроверки

  1. Что такое training-serving skew и как его предотвратить?

    Training-serving skew: разные преобразования признаков при обучении и inference (разный preprocessing, разные версии библиотек). Модель ухудшается в production. Решение: единый feature transformation код (pipeline), feature store, мониторинг распределения признаков в production.

  2. Что такое shadow mode deployment и зачем он нужен?

    Shadow mode: новая модель запускается параллельно со старой, получает те же запросы, но её ответы не используются пользователями. Это позволяет безопасно оценить новую модель на реальном трафике без риска. Сравниваем метрики offline и online перед полным переключением.

  3. Что такое A/B тестирование в контексте ML-моделей?

    A/B тест: случайно делим трафик — 50% видят модель A (baseline), 50% — модель B (новая). Измеряем бизнес-метрики (CTR, конверсию) и ML-метрики (точность, latency). Нужна статистически значимая разница. Длительность: обычно 1-2 недели для снижения variance.

  4. Что такое model monitoring и какие метрики нужно отслеживать?

    Мониторинг включает: (1) drift — изменение распределения входных признаков (data drift) и предсказаний (concept drift), (2) производительность (latency, error rate), (3) качество модели (если есть ground truth labels с задержкой). Triton, Evidently, Arize — инструменты для мониторинга.

  5. When is batch inference preferable to online inference?

    Batch inference is better when: predictions do not need to reflect the very latest context (e.g., daily recommendation refreshes, weekly risk scores), when the volume of items to score is large and cost matters, or when the downstream consumer reads results from a database rather than waiting for a live API call. It is also simpler to operate — no SLO on latency, easier retries, and cheaper hardware.

  6. What is a feature store and why is it needed?

    A feature store is a centralized repository that stores feature definitions, offline historical values, and low-latency online access to the same features. It is needed because without it, the training pipeline and the serving pipeline often compute features using different code, producing slightly different values — train-serve skew. The model was trained on one distribution and serves on another, causing silent quality loss.

  7. How do you detect model degradation in production?

    Use multiple signals in parallel: (1) data drift — monitor input feature distributions against the training baseline using PSI or KS test, alert when shift exceeds threshold; (2) prediction drift — watch the distribution of output scores or labels for unusual shifts; (3) delayed labels — when ground truth arrives (clicks, conversions, fraud labels), compute actual model quality and compare to the pre-deployment baseline; (4) business metrics — monitor downstream KPIs like CTR, revenue, or complaint rate for unexplained drops.

What breaks without this

Training features computed differently from serving features causes train-serve skew — the model sees a different input distribution at inference than it was trained on. It will silently underperform with no error raised, often degrading by 5–15% in real metric before anyone notices.

Как закрепить тему: Паттерны проектирования ML-систем

Блок связывает теорию с практикой: где применять идею, какие ошибки проверять и как развиваться дальше.

Когда применять

  • Feature store — инвестируй в него при работе с несколькими моделями в production.
  • Shadow mode перед A/B тестом — проверь новую модель без риска.
  • Мониторинг drift — обязателен для любой production модели.

Типичные ошибки

  • Не разделять training pipeline и serving pipeline — training-serving skew.
  • Пропускать shadow mode и переходить сразу к A/B тесту.
  • Игнорировать latency SLO при выборе архитектуры модели.

Что тюнить / проверить

  • Confidence threshold для human-in-the-loop сценариев.
  • Canary rollout percentage: 1% → 5% → 25% → 100%.
  • Alert threshold для data drift: обычно PSI > 0.2 — проблема.
