9.1 Experiment Tracking

Эксперимент без логов — это не эксперимент, а воспоминание. Tracking нужен, чтобы через неделю понять, какие данные, seed, hyperparameters, commit, метрики и артефакты дали конкретный результат.

Почему важно

Без experiment tracking ML — это хаос. MLflow, W&B, DVC позволяют воспроизвести любой прогон, сравнить метрики и отследить версии данных. Разница между «у меня где-то лучшая модель» и «вот она».

Главная идея

В ML мы редко запускаем одну модель один раз. Обычно есть десятки runs: разные признаки, learning rate, архитектуры, seed, split данных и preprocessing. Если это не записывать автоматически, невозможно честно сравнить модели и повторить лучший результат.

run = code_version + data_version + config + random_seed + metrics + artifacts

Практический смысл: каждый run должен быть восстановимым объектом. Если ты не можешь ответить “какой код и какие данные породили эту метрику?”, результат нельзя считать надежным.

Что логировать обязательно

Config
Learning rate, batch size, optimizer, scheduler, model size, preprocessing, augmentation, split strategy.
Metrics
Train/validation loss, target metrics, latency, memory, calibration, per-class quality when relevant.
Artifacts
Model checkpoint, tokenizer, feature schema, plots, confusion matrix, predictions sample, error cases.
Environment
Git commit, package versions, CUDA version, seed, data snapshot, hardware and runtime.

MLflow и W&B

MLflow удобен как open-source tracking server и model registry. Weights & Biases особенно силен для dashboards, sweeps, командного сравнения графиков и rich media.

Когда выбирать MLflow

Нужен self-hosted контроль, registry, локальное хранение или интеграция в enterprise-инфраструктуру.

Когда выбирать W&B

Нужны удобные визуализации, sweeps, командная работа и быстрый анализ больших серий экспериментов.

Мини-трекер без зависимостей

# Minimal experiment tracker.
# Input: config dict and metric dicts.
# Output: runs/<run_id>/config.json and metrics.jsonl.
import json
import time
from pathlib import Path

class RunLogger:
    def __init__(self, config, root="runs"):
        self.run_id = time.strftime("%Y%m%d-%H%M%S")
        self.dir = Path(root) / self.run_id
        self.dir.mkdir(parents=True, exist_ok=False)
        (self.dir / "config.json").write_text(
            json.dumps(config, indent=2, ensure_ascii=False),
            encoding="utf-8",
        )
        self.metrics_path = self.dir / "metrics.jsonl"

    def log(self, step, **metrics):
        row = {"step": step, **metrics}
        with self.metrics_path.open("a", encoding="utf-8") as f:
            f.write(json.dumps(row) + "\n")

logger = RunLogger({"lr": 1e-3, "batch_size": 64, "seed": 42})
for step in range(3):
    logger.log(step, train_loss=1.0 / (step + 1), val_acc=0.70 + step * 0.03)

# Gotchas:
# 1. Do not overwrite run directories.
# 2. Log config before training starts.

Как сравнивать runs

Сравнивать нужно не только лучший validation score. Хороший run должен быть стабильным по seed, иметь приемлемую latency, не переобучаться, не ломаться на важных подгруппах данных и быть воспроизводимым.

model_selection = quality + stability + latency + reproducibility + operational_cost

Пошагово What to log in every run
1 /
Step 01

Log config at run start

Before the first training step, log every hyperparameter: learning rate, batch size, optimizer name, model architecture string, data path, random seed, and train/val split strategy. This snapshot defines what this run is.

Step 02

Log metrics every N steps

During training, record train loss, validation loss, and your primary target metric at a fixed interval (e.g., every 100 steps or every epoch). Logging only the final value hides instability and prevents diagnosing overfitting.

Step 03

Log artifacts on improvement

Whenever validation metric improves, save the model checkpoint, tokenizer, and feature schema. Log them as versioned artifacts so you can always recover the exact model that produced a specific metric value.

Step 04

Log environment

At the end of the run, record the git commit hash, a snapshot of requirements.txt, CUDA version, and hardware info (GPU model, memory). Without the environment snapshot, even identical code and config may not reproduce the same result.

Проверь себя

Вопросы для самопроверки

  1. Что такое experiment tracking и зачем он нужен?

    Experiment tracking — запись параметров, метрик, артефактов и кода каждого эксперимента. Без него невозможно воспроизвести лучший результат, сравнить модели или объяснить коллеге, что именно работает. MLflow, W&B, Neptune — популярные инструменты.

  2. Что должны включать метаданные каждого ML-эксперимента?

    Минимум: гиперпараметры, версия данных (хеш), git commit, метрики по эпохам, итоговые тест-метрики, время обучения. Желательно: артефакты (модель, логи), окружение (requirements.txt), случайные seed, конфигурационный файл. Это гарантирует полную воспроизводимость.

  3. Что такое model registry и чем он отличается от artifact store?

    Artifact store хранит все артефакты (модели, данные) без структуры. Model registry добавляет жизненный цикл: стадии (staging, production, archived), версионирование, метаданные о deployment. Это позволяет управлять «живыми» моделями в production и контролировать переходы между версиями.

  4. Как организовать сравнение экспериментов при тюнинге гиперпараметров?

    Фиксируй случайный seed и данные (одинаковые splits). Меняй один параметр за раз или используй structured search (grid/random/Bayesian). В W&B/MLflow используй sweep/group для визуализации корреляций параметр-метрика. Логируй validation metric на каждой эпохе, не только финальный.

  5. What is the minimum information needed to reproduce a run?

    You need: the exact code version (git commit hash), data version or data path with a content hash, full config (all hyperparameters and random seeds), environment snapshot (package versions, CUDA version), and the saved model artifact. Missing any one of these makes reproduction unreliable.

  6. What breaks reproducibility even with the same code and config?

    Non-deterministic GPU operations (cuDNN autotuning, multi-threaded data loading without fixed workers), different package versions (even minor updates can change numerics), different data ordering if shuffling is not seeded, and different hardware (different GPU architectures may produce slightly different floating point results).

  7. When should you log a model artifact vs just metrics?

    Log a model artifact whenever validation metric improves — you need the checkpoint to actually use or compare that model later. Logging only metrics tells you a run was good but gives you nothing to deploy or inspect. Also log artifacts at the end of training regardless of improvement, so you can always restart from a known state.

What breaks without this

Skipping git commit logging means you cannot reproduce results after any code change — even one refactor or dependency update makes the run permanently unreproducible, because there is no way to recover which version of the code produced that metric.

Как закрепить тему: Отслеживание экспериментов

Блок связывает теорию с практикой: где применять идею, какие ошибки проверять и как развиваться дальше.

Когда применять

  • Логируй каждый запуск с самого начала — восстановить историю задним числом невозможно.
  • Используй config files (YAML/JSON) + логирование всего config в трекер.
  • W&B sweeps или MLflow Tracking + Optuna — мощная связка для гиперпараметрического поиска.

Типичные ошибки

  • Не сохранять random seed — результат невоспроизводим.
  • Логировать только финальную метрику без кривых обучения.
  • Смешивать staging и production модели в одном registry.

Что тюнить / проверить

  • Число запускаемых параллельно экспериментов — зависит от GPU бюджета.
  • Метрики для сравнения: определяй заранее, не после просмотра результатов.
  • Retention policy для артефактов — модели занимают место.
